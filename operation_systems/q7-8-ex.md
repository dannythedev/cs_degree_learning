# סיכום קורס מערכות הפעלה (20594) - סנכרון תהליכים (שאלה 7) באמצעות אסוציאציות

כדי לפתור את בעיות הסנכרון בלי להתבלבל, נשתמש בשפה פשוטה ואחידה. תשכח ממונחים מסובכים, זה המילון שלנו מעכשיו:

## המילון הקצר (הכלים)
*   **`down` = "לקחת" 🖐️:** אני מנסה לקחת משהו. אם הקופסה ריקה? אני נופל על הרצפה והולך לישון עד שמישהו ישים שם משהו.
*   **`up` = "לתת" 🤲:** אני מחזיר משהו או שם משהו חדש בקופסה. הפעולה הזו מעירה מיד את מי שישן ומחכה.
*   **`mutex` = "מפתח לשירותים" 🔑:** יש רק מפתח אחד כזה בעולם. אם תפסתי אותו (`down`), נעלתי את הדלת ואף אחד לא יכול להיכנס. כשסיימתי, אני מחזיר אותו (`up`).
*   **`empty` / `full` = "קערת אסימונים" 🪙:** קופסאות שיש בהן הרבה כרטיסים. למשל, כרטיסי "מקום פנוי בחניה" (`empty`) או כרטיסי "מוצר מוכן" (`full`).

## 3 חוקי הברזל (מתי שמים מה)
1.  **חוק ה"סנדוויץ'" (הגנה על משתנים):** אם אתה עושה פלוס-פלוס (`++`) או מינוס-מינוס (`--`) למספר, אתה **חייב** לקחת קודם את ה"מפתח לשירותים" (`down(mutex)`), לעשות את החישוב בפרטיות, ולהחזיר את המפתח (`up(mutex)`).
2.  **חוק "קודם חניה, אחר כך מפתח" (למניעת Deadlock):** כשיש לך שני `down` רצופים (אחד לאסימון ואחד לשירותים), **תמיד** קח קודם את האסימון של החניה (`down(empty/full)`), ורק אז את המפתח לשירותים (`down(mutex)`). אם תעשה הפוך - תנעל את הדלת, תגלה שאין כלום, תירדם, ואף אחד לא יוכל להיכנס לעזור לך.
3.  **חוק "הראשון מדליק אור, האחרון מכבה" (לקוראים-כותבים):** כשקבוצה (כמו קוראים) עובדת יחד, רק הראשון שנכנס (`if count == 1`) אחראי לנעול את האויב. רק האחרון שיוצא (`if count == 0`) אחראי לשחרר את האויב. השאר סתם נכנסים ויוצאים.

## 10 תרגילים מדורגים (מאפס לרמת מבחן)

### תרגיל 1: ההגנה הבסיסית (חוק הסנדוויץ')
**השאלה:** יש משתנה משותף `Counter=0`. תהליך א' ותהליך ב' עושים `Counter++` יחד. השלם את הקוד כדי שהתוצאה לא תיהרס. נתון: `mutex=1`.
**סיפור הדמיה:** שני תלמידים רוצים לכתוב את התוצאה החדשה על הלוח. אם שניהם יכתבו יחד, זה יצא קשקוש.
**פתרון:** נשתמש ב"מפתח לשירותים" (חוק הסנדוויץ'). רק מי שעם המפתח יכול לכתוב.
```c
Process:
1. down(mutex); // לוקח 🖐️ מפתח לשירותים
2. Counter++;   // פעולה אגרסיבית! (מוגנת)
3. up(mutex);   // נותן 🤲 את המפתח חזרה
```

---

### תרגיל 2: פינג-פונג (המתנה לאישור)
**השאלה:** תהליך א' אומר "Ping", תהליך ב' אומר "Pong". איך נבטיח שתמיד "Ping" יודפס קודם? נתון: `sync=0`.
**סיפור הדמיה:** תהליך ב' לא יכול לעשות כלום עד שתהליך א' ייתן לו "אסימון". 
**פתרון:** 
```c
Process A (Ping):
1. print("Ping");
2. up(sync);   // נותן 🤲 אסימון שיעיר את B

Process B (Pong):
1. down(sync); // מנסה לקחת 🖐️ אסימון. אם A לא נתן, B יירדם פה!
2. print("Pong");
```

---

### תרגיל 3: יצרן-צרכן עם מחסן אינסופי
**השאלה:** יצרן שם מוצרים, צרכן לוקח. המחסן ענק (אין גבול למקומות). איך נדאג שהצרכן לא ייקח כשהמחסן ריק?
**סיפור הדמיה:** יש רק "קערת אסימונים" אחת בשם `items` (שמתחילה ב-0). היצרן שם אסימונים, הצרכן לוקח.
**פתרון:**
```c
Producer:
1. up(items);   // היצרן נותן 🤲 אסימון "יש מוצר!"

Consumer:
1. down(items); // הצרכן מנסה לקחת 🖐️ אסימון "יש מוצר". אם אין, נרדם.
```

---

### תרגיל 4: יצרן-צרכן (הקלאסי של המבחן - באפר מוגבל)
**השאלה:** המחסן מוגבל ל-N מקומות. היצרן עוצר אם מלא. הצרכן עוצר אם ריק. מותר רק לאחד לגעת במחסן בכל רגע. נתונים: `empty=N` (קערת אסימוני חניה), `full=0` (קערת אסימוני מוצרים), `mutex=1` (מפתח לשירותים).
**סיפור הדמיה:** היצרן חייב קודם למצוא חניה (`empty`), ואז לקחת את המפתח לשירותים (`mutex`), לשים את המוצר, להחזיר את המפתח, ולשים אסימון בקערת ה"מוצר מוכן" (`full`).
**פתרון:** חוק "קודם חניה, אחר כך מפתח"!
```c
Producer:
1. down(empty); // 🖐️ לקחת אסימון חניה
2. down(mutex); // 🖐️ לקחת מפתח לשירותים
3. Enqueue();   // לשים את המוצר
4. up(mutex);   // 🤲 להחזיר מפתח לשירותים
5. up(full);    // 🤲 לתת אסימון "מוצר מוכן"

Consumer:
1. down(full);  // 🖐️ לקחת אסימון "מוצר מוכן"
2. down(mutex); // 🖐️ לקחת מפתח לשירותים
3. Dequeue();   // לקחת את המוצר
4. up(mutex);   // 🤲 להחזיר מפתח לשירותים
5. up(empty);   // 🤲 לתת אסימון "התפנתה חניה"
```

---

### תרגיל 5: מלכודת ה-Deadlock (שאלת הבנה)
**השאלה:** מה יקרה אם הצרכן בתרגיל 4 יהפוך את שורות 1 ו-2 (יעשה `down(mutex)` ואז `down(full)`)?
**סיפור הדמיה:** הצרכן לקח את ה"מפתח לשירותים" ונעל את הדלת מבומנים. עכשיו הוא ניגש ל"קערת המוצרים המוכנים", רואה שהיא ריקה, ו**נרדם** בתוך השירותים עם המפתח ביד.
היצרן בא עם מוצר חדש, אבל הוא לא יכול להיכנס כי הדלת נעולה. 
**התשובה:** קיפאון נצחי (Deadlock). הפרנו את חוק "קודם חניה, אחר כך מפתח".

---

### תרגיל 6: קוראים וכותבים - הבסיס (חוק האור והחושך)
**השאלה:** הרבה קוראים יכולים לקרוא יחד. כותב חייב להיות לבד.
**סיפור הדמיה:** יישום חוק "הראשון מדליק אור, האחרון מכבה". התלמיד (הקורא) הראשון שנכנס נועל את ה"אויב" (הכותב) בחוץ. כל השאר פשוט נכנסים. כשהתלמיד האחרון יוצא, הוא פותח לכותב. כמובן שספירת התלמידים דורשת חוק "סנדוויץ'"!
**פתרון:**
```c
Reader:
1. down(RMutex);          // 🖐️ מפתח לשירותים בשביל הלוח-ספירה
2. Readers++;             // רושם את עצמי
3. if (Readers == 1)      // הראשון! מדליק אור (נועל אויבים)
4.    down(BlockWrite);   // 🖐️ לוקח לכותב את המפתח
5. up(RMutex);            // 🤲 מחזיר את מפתח הלוח-ספירה

< קורא מהספר בנחת >

6. down(RMutex);          // 🖐️ מפתח לשירותים ללוח-ספירה
7. Readers--;             // מוחק את עצמי
8. if (Readers == 0)      // האחרון! מכבה אור
9.    up(BlockWrite);     // 🤲 מחזיר לכותב את המפתח
10. up(RMutex);           // 🤲 מחזיר את מפתח הלוח-ספירה
```

### תרגיל 7: קוראים וכותבים עם עדיפות לכותב (רמת מבחן!)
**השאלה:** בדיוק כמו תרגיל 6, אבל עכשיו אנחנו שמים "שומר בשער" (`BlockRead`). כשהכותב מגיע, הוא נועל את השומר כדי שאף קורא *חדש* לא יוכל להיכנס. 
**סיפור הדמיה:** הכותב הראשון שמגיע לתור נועל מיד את השער החיצוני. קוראים חדשים ייתקעו כבר בצעד הראשון שלהם.
**פתרון ל-Writer:** מפעילים את חוק "הראשון והאחרון" גם על הכותבים!
```c
Writer:
1. down(WMutex);          // 🖐️ מפתח לשירותים ללוח-ספירת-כותבים
2. Writers++;
3. if (Writers == 1)      // הכותב הראשון בתור!
4.    down(BlockRead);    // 🖐️ לוקח לקוראים את האפשרות לעבור בשער
5. up(WMutex);            // 🤲 מחזיר מפתח ספירה

6. down(BlockWrite);      // 🖐️ מחכה שהספר עצמו יתפנה
7. < כותב בספר >
8. up(BlockWrite);        // 🤲 משחרר את הספר

9. down(WMutex);          // סנדוויץ' סיום
10. Writers--;
11. if (Writers == 0)     // הכותב האחרון שעוזב
12.    up(BlockRead);     // 🤲 פותח חזרה את השער לקוראים
13. up(WMutex);
```

### תרגיל 8: מפעל הממתקים (ריבוי יצרנים)
**השאלה:** יש 10 מכונות שמייצרות ממתקים לאותה קופסה (בגודל N). מה חובה להוסיף לעומת יצרן רגיל?
**סיפור הדמיה:** עכשיו יש 10 שפים. אוקיי, כולם יכולים לקחת אסימון מ"קערת החניה" (`empty`), אבל כדי להניח את הסושי על המסוע הם חייבים את ה"מפתח לשירותים" (`mutex`), אחרת הידיים שלהם יתנגשו וימעכו את האוכל.
**פתרון:** אין שום שינוי בקוד מתרגיל 4! ה-`down(mutex)` ו-`up(mutex)` שעוטפים את פעולת ה-`Enqueue` הם אלו שמצילים את המצב במקרה של ריבוי יצרנים.

### תרגיל 9: זיהוי סמפור עלום (מבחן)
**השאלה:** נתון קוד של צרכן: 
`down(S1);` 
`item = Dequeue();` 
`up(S1);`
מהו `S1`? 
**סיפור הדמיה:** מה זה ה-`down` וה-`up` שעוטפים פעולה רגישה? זה בדיוק ה"מפתח לשירותים"!
**פתרון:** זהו סמפור בינארי (מיוטקס). תפקידו לספק "מניעה הדדית" (Mutual Exclusion) על מנהל התור, כלומר למנוע משני צרכנים למשוך נתונים בדיוק באותה המילי-שנייה ולהרוס את הזיכרון. זהו חוק הסנדוויץ'.

### תרגיל 10: בחירה בין מחסנים (Try_down)
**השאלה:** היצרן רוצה לשים מוצר במחסן A. אם אין מקום ב-A, הוא ישים ב-B. הוא אסור לו לישון ליד A אם A מלא ו-B ריק.
**סיפור הדמיה:** אני מגיע עם רכב. אני בודק אם יש "אסימון חניה" בקערה A. **אני רק מסתכל**, אני לא מנסה "לקחת בכוח" כי אז אני ארדם שם. אם יש – אני לוקח. אם לא, אני בודק את קערה B.
**פתרון:** נשתמש בתנאי רגיל לפני פעולת הלקיחה, כדי לא להיתקע בחוק הראשון (שמרדים אותנו).
```c
Producer:
// בודק אם "קערת החניה" של A גדולה מ-0:
if (emptyA > 0) {
    down(emptyA); // עכשיו בטוח לקחת בלי להירדם
    down(mutexA);
    EnqueueA();
    up(mutexA);
    up(fullA);
} 
// אם A מלא, ננסה את B:
else {
    down(emptyB); // פה מותר לי להירדם אם גם B מלא
    down(mutexB);
    EnqueueB();
    up(mutexB);
    up(fullB);
}
```

# סיכום קורס מערכות הפעלה (20594) - ניהול זיכרון ותרגום כתובות (שאלה 8) באסוציאציות

הנושא הזה נראה כמו שיעור מסובך במתמטיקה בינארית, אבל בפועל מדובר רק ב"משחק של כתובות וארגזים". נלמד את המושגים דרך סיפור פשוט כדי שהכל "ידבק".

## המילון הקצר (הכלים והסיפור שלנו)
דמיין שאתה מנהל ענק של "מחסני אמזון".
*   **זיכרון וירטואלי (Virtual Memory) = "הכתובת המדומה":** הכתובת שהלקוח (התוכנה) חושב שיש לו. הלקוח בטוח שיש לו מחסן פרטי ענק משלו.
*   **זיכרון פיזי (RAM / Physical Memory) = "המחסן האמיתי":** הבניין הפיזי שבו אנחנו באמת מאחסנים דברים. הוא הרבה יותר קטן מהמחסן המדומה.
*   **דף (Page) = "ארגז קרטון":** אנחנו לא מעבירים פריטים בודדים, אלא אורזים הכל בארגזים בגודל קבוע (למשל, ארגז שמכיל 4,096 פריטים). זה הדף הווירטואלי.
*   **מסגרת (Frame) = "מדף במחסן":** המקום האמיתי ב-RAM. **חוק ברזל:** הגודל של המדף שווה בדיוק לגודל של ארגז הקרטון! ארגז אחד נכנס בדיוק למדף אחד.
*   **היסט (Offset) = "המיקום בתוך הארגז":** אם אתה מחפש את הפריט ה-15 בתוך הארגז, זה לא משנה אם הארגז נמצא באילת או בחיפה – הוא תמיד יהיה הפריט ה-15 בארגז הספציפי הזה. **מסקנה: ה-Offset לעולם לא משתנה בתרגום!**
*   **טבלת דפים (Page Table) = "ספר האינדקס":** הספר שאומר למנהל העבודה: "ארגז מספר 2 של הלקוח נמצא כרגע על מדף מספר 50 במחסן האמיתי". 
*   **PTE (Page Table Entry) = "שורה בספר האינדקס":** שורה אחת שאומרת איפה הארגז. יש בה גם **Valid Bit** ("ביט נוכחות"): 1 = הארגז במחסן האמיתי (RAM). 0 = הארגז נשאר במפעל מרוחק (דיסק קשיח) וצריך להביא אותו. הבאה כזו נקראת **Page Fault** (פסיקת דף).

## 3 חוקי הברזל לחישובים במבחן
1.  **חוק החזקות של 2 (שפת המחשב):**
    *   $1KB = 2^{10}$ בתים.
    *   $4KB = 2^{12}$ בתים (גודל הארגז/דף הנפוץ ביותר במבחן).
    *   $1MB = 2^{20}$ בתים.
    *   $1GB = 2^{30}$ בתים.
2.  **חוק האופסט (Offset):** אם גודל הדף הוא $2^k$, ה-Offset הוא תמיד **$k$ הביטים הימניים ביותר** של הכתובת. השאר (הביטים השמאליים) הם מספר הדף.
3.  **חוק "טבלה בגודל דף":** במבחן יבקשו שטבלת האינדקס תיכנס בדיוק בארגז אחד. כדי לדעת כמה שורות נכנסות לארגז, עושים:
    `מספר שורות = (גודל דף) חלקי (גודל של שורה אחת PTE)`.

## 10 תרגילים מדורגים (מאפס לרמת מבחן)

### תרגיל 1: מציאת ה-Offset (בסיס)
**השאלה:** נתון מערכת עם דפים בגודל $4KB$. כמה ביטים צריך עבור ה-Offset בכתובת?
**סיפור הדמיה:** בארגז נכנסים 4,096 פריטים. כמה ספרות (בינאריות) אתה צריך כדי למספר כל פריט בארגז מ-0 ועד 4,095?
**פתרון:** 
נמיר את גודל הדף לבתים ולחזקה של 2: $4KB = 4 \times 1024 = 4096 = 2^{12}$.
המעריך הוא מספר הביטים. **תשובה: 12 ביט.**

---

### תרגיל 2: פירוק כתובת בינארית לשניים
**השאלה:** נתונה כתובת וירטואלית של 16 ביט: `001100 0000000101`. גודל דף הוא $1KB$. מהו מספר הדף ומהו האופסט (המיקום בתוך הדף)?
**סיפור הדמיה:** קיבלת קוד ברקוד ארוך. החלק הימני אומר "איזה פריט בארגז", והחלק השמאלי אומר "איזה ארגז". איפה עובר החיתוך?
**פתרון:**
1. גודל דף $1KB = 2^{10}$. לכן, ה-Offset הוא **10 הביטים הימניים**.
2. נחתוך את הכתובת: `001100` (שמאל) | `0000000101` (ימין).
3. המספר השמאלי הוא מספר הדף: `001100` (שזה 12 בעשרונית).
4. המספר הימני הוא האופסט: `0000000101` (שזה 5 בעשרונית).
**תשובה: דף 12, אופסט 5.**

---

### תרגיל 3: תרגום כתובת פשוט (עשרוני)
**השאלה:** כתובת וירטואלית היא **10,000**. גודל דף הוא $4,096$ בתים. בטבלת הדפים רשום שדף 2 נמצא במסגרת 5. מהי הכתובת הפיזית?
**סיפור הדמיה:** הלקוח שואל: "איפה פריט מספר 10,000 שלי?". אתה מחשב באיזה ארגז הוא נמצא, בודק בספר האינדקס איפה הארגז מונח במחסן האמיתי, ומחשב את המיקום המדויק.
**פתרון:**
1. איזה ארגז (דף)? $10,000 / 4,096 = 2.44$. כלומר, אנחנו ב**דף 2**.
2. איזה פריט בארגז (Offset)? השארית. $10,000 \pmod{4096} = 1,808$.
3. תרגום למחסן האמיתי: האינדקס אומר שדף 2 נמצא ב**מסגרת 5**. תחילת מסגרת 5 היא בכתובת $5 \times 4,096 = 20,480$.
4. נוסיף את הפריט שבתוך הארגז (האופסט לא משתנה!): $20,480 + 1,808 = 22,288$.
**תשובה: 22,288.**

---

### תרגיל 4: זיהוי Page Fault (מלכודת במבחן)
**השאלה:** נתונה טבלת דפים. עבור דף 3 רשום: `Frame = 8, Valid Bit = 0`. הכתובת הווירטואלית המבוקשת שייכת לדף 3. מהי הכתובת הפיזית?
**סיפור הדמיה:** אתה פותח את ספר האינדקס. רשום בו: "הארגז הזה לא כאן (Valid=0), הוא נשאר במפעל באילת (הדיסק הקשיח)".
**פתרון:**
לעולם אל תחשב כתובת פיזית אם ה-Valid Bit הוא 0! זה אומר שהדף לא ב-RAM.
**תשובה:** תתרחש פסיקת דף (Page Fault). מערכת ההפעלה תעצור הכל ותביא את הדף מהדיסק.

### תרגיל 5: למה בכלל צריך טבלה רב-שלבית? (שאלת הבנה)
**השאלה:** במערכת 32 ביט עם דפים של $4KB$, כמה דפים יש לתהליך מקסימלי, וכמה מקום תתפוס טבלת דפים רגילה (שטוחה) אם כל שורה (PTE) שוקלת 4 בתים?
**סיפור הדמיה:** אם ניתן לכל לקוח "ספר אינדקס" שלם שכולל את כל הארגזים האפשריים בעולם, כמה שוקל הספר הזה?
**פתרון:**
1. כתובת 32 ביט = מרחב של $4GB = 2^{30}$ בתים.
2. גודל דף $4KB = 2^{12}$ בתים.
3. כמות דפים אפשרית: $2^{30} / 2^{12} = 2^{18}$ דפים (בערך רבע מיליון דפים).
4. משקל הטבלה: $2^{18} \times 4 \text{ bytes} = 2^{20} \text{ bytes} = 1MB$.
**הבנה למבחן:** 1MB לכל תהליך זה המון בזבוז אם התהליך קטן! לכן עברו לטבלאות רב-שלביות, כדי לא לשמור אינדקסים לארגזים ריקים.

### תרגיל 6: חלוקת כתובת לטבלה דו-שלבית (הקלאסי של המבחן!)
**השאלה:** מערכת 32 ביט. גודל דף $4KB$. גודל PTE הוא 4 בתים. תכנן חלוקה לטבלה דו-שלבית (Level 1, Level 2, Offset) כך ש**כל טבלה תתפוס בדיוק דף אחד**.
**סיפור הדמיה:** מנהל המחסן אומר: "ספר האינדקס הראשי, וכל ספרי האינדקס המשניים, חייבים להיכנס בדיוק לארגז אחד סטנדרטי (4KB)".
**פתרון:**
1. **Offset:** גודל דף $4KB = 2^{12}$ $\leftarrow$ צריך **12 ביט**.
2. **כמה שורות נכנסות בדף (ארגז)?**
   גודל הדף הוא 4,096 בתים. כל שורה שוקלת 4 בתים. 
   $4,096 / 4 = 1,024$ שורות נכנסות בטבלה.
3. **ביטים לאינדקס:** כדי למספר 1,024 שורות, צריך $2^{10}$ $\leftarrow$ **10 ביט**.
4. **מסקנה:** רמה 1 צריכה 10 ביט. רמה 2 צריכה 10 ביט. האופסט 12 ביט.
5. בדיקה: $10+10+12 = 32$. מתאים לנתון!
**תשובה: 10-10-12.** *(טיפ: זה הפתרון ל-90% משאלות החלוקה במבחנים!)*

### תרגיל 7: חלוקת כתובת עם נתונים מוזרים (רמת מבחן גבוהה)
**השאלה:** מערכת 32 ביט. גודל PTE הוא 4 בתים. נתון שהטבלה היא דו-שלבית, ורמה 1 (Level 1) מקבלת **8 ביט**, רמה 2 מקבלת **8 ביט**. מה חייב להיות גודל הדף?
**סיפור הדמיה:** עכשיו עובדים הפוך. אתה יודע כמה ספרות יש בתווית, אבל לא יודע מה גודל הארגז.
**פתרון:**
1. כתובת מלאה = רמה 1 + רמה 2 + אופסט.
2. $32 = 8 + 8 + \text{Offset}$.
3. $\text{Offset} = 32 - 16 = 16$.
4. גודל הדף הוא תמיד 2 בחזקת האופסט: $2^{16}$ בתים.
5. $2^{16} = 64 \times 1024 = 64KB$.
**תשובה: גודל הדף הוא 64KB.**

### תרגיל 8: כמה דפים תופסת הטבלה עצמה? (רמת מבחן - מתוך PDF)
**השאלה:** במערכת עם חלוקה של 10-10-12 (32 ביט, דפים של 4KB), כמה דפים ב-RAM יתפסו **כל** טבלאות הדפים יחד (ראשית ומשניות), בהנחה שהתהליך מנצל את **כל** הזיכרון הווירטואלי שלו?
**סיפור הדמיה:** הלקוח החליט למלא את כל המחסן המדומה שלו! זה אומר שאנחנו חייבים להדפיס לו את כל ספרי האינדקס המשניים. כמה "ארגזים" יתפסו הספרים עצמם?
**פתרון:**
1. **טבלה ראשית (Level 1):** תמיד יש אחת כזו. תופסת **1 דף**.
2. **טבלאות משניות (Level 2):** הטבלה הראשית בנויה מ-10 ביט, כלומר יש בה $2^{10} = 1,024$ שורות. בגלל שהלקוח מנצל את כל הזיכרון, כל שורה בטבלה הראשית חייבת להצביע לטבלה משנית אמיתית.
3. לכן נצטרך **1,024 טבלאות משניות**. כל טבלה כזו תופסת דף אחד (לפי התכנון).
4. סה"כ: 1 (ראשית) + 1,024 (משניות) = **1,025 דפים**.

---

### תרגיל 9: תהליך קטן במערכת דו-שלבית (שאלת הבנה עמוקה)
**השאלה:** באותה מערכת של 10-10-12, הרצנו תהליך קטנטן שדורש רק **12MB** של קוד ונתונים (כלומר הוא צריך רק 3,072 דפים). כמה טבלאות דפים משניות נצטרך ליצור עבורו?
**סיפור הדמיה:** הלקוח שכר רק חלק קטן מהמחסן. אנחנו לא נדפיס לו ספרי אינדקס מיותרים למדפים שהוא לא משתמש בהם.
**פתרון:**
1. כל טבלה משנית מנהלת 1,024 דפים (כפי שראינו, יש בה $2^{10}$ כניסות).
2. הלקוח צריך 3,072 דפים סה"כ.
3. כמה "ספרי אינדקס משניים" יכסו 3,072 דפים? $3,072 / 1,024 = \mathbf{3}$.
4. לכן ניצור עבורו טבלה ראשית אחת (1 דף) ורק 3 טבלאות משניות (3 דפים).
5. סה"כ 4 דפים לניהול במקום 1,025. זה הקסם של טבלה דו-שלבית!

### תרגיל 10: זיהוי סוג התרגום (TLB - מושג משלים למבחן)
**השאלה:** מהו ה-TLB (Translation Lookaside Buffer) ולמה צריך אותו אם יש לנו טבלת דפים מושלמת ב-RAM?
**סיפור הדמיה:** "הזיכרון הקצר של הספרנית". בכל פעם שהסטודנט מבקש ארגז, הספרנית הולכת לטבלה המרכזית ב-RAM (וזה איטי). אם הסטודנט מבקש את ארגז 5, ואחרי שנייה מבקש שוב את ארגז 5 – היא תלך שוב לטבלה? לא! היא תרשום לה על פתק בכיס: "ארגז 5 זה מסגרת 20".
**פתרון:** 
ה-TLB הוא Cache (מטמון) סופר-מהיר בתוך המעבד ששומר את הכתובות שתורגמו לאחרונה. כשכתובת מגיעה, בודקים קודם ב-TLB (נקרא TLB Hit). רק אם היא לא שם (TLB Miss), הולכים לטבלת הדפים ב-RAM לעשות את התרגום הארוך. זה חוסך המון זמן.
