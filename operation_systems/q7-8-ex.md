# סיכום קורס מערכות הפעלה (20594) - סנכרון תהליכים (שאלה 7)

נושא הסנכרון הוא "עמוד השדרה" של המבחן. הוא עוסק בשאלה: איך גורמים למספר תהליכים לעבוד יחד על אותו משאב מבלי ליצור בלגן, השחתת נתונים או תקיעה של המערכת.

## 1. מושגי יסוד - השפה של הסנכרון

כדי להבין את הפתרונות, אני חייב לשלוט במושגים האלו כאילו הם שפת אם:

*   **מרוץ קריטי (Race Condition):** המצב שבו שני תהליכים משנים משתנה יחד והתוצאה תלויה ב"מי הגיע קודם". זה מצב רע שחייבים למנוע.
*   **קטע קריטי (Critical Section):** שורות הקוד שבהן ניגשים למשאב המשותף.
*   **מניעה הדדית (Mutual Exclusion):** החוק שאומר: "רק אחד בתוך הקטע הקריטי בכל רגע".
*   **סמפור (Semaphore):** "קופסת מפתחות".
    *   **`down` (או P / wait):** פעולה של לקחת מפתח. אם אין מפתחות בקופסה – התהליך נרדם ומחכה.
    *   **`up` (או V / signal):** פעולה של להחזיר מפתח לקופסה. אם מישהו חיכה, הוא מתעורר מיד.
*   **מיוטקס (Mutex):** סמפור בינארי. יש בו רק מפתח **אחד**. משמש כ"מנעול" (Lock).

---

## 2. מאגר תרגילים לתרגול (מרמת בסיס עד רמת מבחן)

### תרגיל 1: הגנה על משתנה (מיוטקס פשוט)
**השאלה:** שני תהליכים רוצים להגדיל משתנה בשם `Counter`. השלם את הקוד כך שלא ייווצר מרוץ קריטי.
**נתון:** `mutex = 1`.

**סיפור הדמיה: "השלט רחוק" 📺**
יש טלוויזיה אחת ושלט אחד. מי שמחזיק בשלט יכול להעביר ערוץ. כדי להגדיל את ה-Counter, תהליך חייב לתפוס את השלט (`down`), לשנות ערוץ, ולהחזיר את השלט לשולחן (`up`).
**פתרון:**
```c
Process:
1. down(mutex); // תופס את השלט
2. Counter++;   // משנה ערוץ
3. up(mutex);   // מחזיר את השלט
```

---

### תרגיל 2: תיאום פשוט (אסימונים)
**השאלה:** יצרן מייצר מוצרים ושם אותם במחסן אינסופי. צרכן לוקח מוצרים. השלם את הקוד כך שהצרכן לא ינסה לקחת מוצר כשאין כלום.
**נתון:** `items = 0` (סופר כמה מוצרים יש).

**סיפור הדמיה: "מכונת המסטיקים" 🍬**
כל פעם שהיצרן שם מסטיק במכונה, הוא מוסיף אסימון לקערה (`up`). הילד (הצרכן) ניגש לקערה. אם יש אסימון – הוא לוקח אותו (`down`) ומוציא מסטיק. אם אין אסימון – הוא יושב ליד המכונה ומחכה.
**פתרון:**
```c
Producer:
1. <מייצר מוצר>
2. up(items);   // מוסיף אסימון

Consumer:
1. down(items); // מחכה שיהיה אסימון
2. <לוקח מוצר>
```

---

### תרגיל 3: יצרן-צרכן עם מחסן מוגבל (N)
**השאלה:** המחסן יכול להכיל רק **N** פריטים. היצרן לא יכול להכניס אם המחסן מלא. הצרכן לא יכול להוציא אם הוא ריק.
**נתון:** `mutex=1`, `empty=N` (מקומות פנויים), `full=0` (מוצרים מוכנים).

**סיפור הדמיה: "המסוע בסושייה" 🍣**
המסוע יכול להכיל רק 10 צלחות (N=10).
*   השף (יצרן) חייב לבדוק שיש מקום ריק על המסוע (`down(empty)`).
*   הלקוח (צרכן) חייב לבדוק שיש צלחת עם אוכל (`down(full)`).
*   בגלל שהנחת צלחת היא פעולה עדינה, רק אחד נוגע במסוע בכל פעם (`mutex`).

**פתרון:**
```c
Producer:
1. down(empty); // יש מקום?
2. down(mutex); // נעל את המסוע
3. Enqueue();   // שים סושי
4. up(mutex);   // שחרר מסוע
5. up(full);    // תגיד ללקוח שיש סושי

Consumer:
1. down(full);  // יש סושי?
2. down(mutex); // נעל את המסוע
3. Dequeue();   // קח סושי
4. up(mutex);   // שחרר מסוע
5. up(empty);   // תגיד לשף שהתפנה מקום
```

---

### תרגיל 4: קוראים-כותבים (בסיס)
**השאלה:** קוראים רבים יכולים לקרוא יחד. כותב חייב להיות לבד. הראשון נועל, האחרון משחרר.
**נתון:** `mutex=1` (הגנה על המונה), `db=1` (מנעול למסד הנתונים), `rcount=0`.

**סיפור הדמיה: "לוח המודעות בכיתה" 📋**
*   **הקוראים (תלמידים):** הרבה יכולים להסתכל יחד. התלמיד הראשון שמגיע נועל את הלוח כדי שהמורה לא ימחק אותו. התלמיד האחרון שעוזב פותח את הנעילה.
*   **הכותב (המורה):** מחכה שכל התלמידים ילכו, נועל את הלוח, כותב ופותח.

**פתרון:**
```c
Reader:
1. down(mutex);
2. rcount++;
3. if (rcount == 1) down(db); // הראשון נועל את הלוח למורה
4. up(mutex);
5. <READ>
6. down(mutex);
7. rcount--;
8. if (rcount == 0) up(db);   // האחרון פותח את הלוח למורה
9. up(mutex);

Writer:
1. down(db); // מחכה שהלוח יהיה ריק ונועל אותו
2. <WRITE>
3. up(db);
```

---

### תרגיל 5: יצרן-צרכן עם שני מחסנים (רמת מבחן)
**השאלה:** יש מחסן A ומחסן B. היצרן מנסה לשים ב-A, ואם הוא מלא – הוא שם ב-B. הצרכן לוקח רק מ-A.
**נתון:** `emptyA, fullA, emptyB, fullB, mutexA, mutexB`.

**סיפור הדמיה: "המוסך והחנייה" 🚗**
יש מוסך (מחסן A) וחניית המתנה (מחסן B). היצרן (גרר) מביא מכונית. אם יש מקום במוסך – הוא שם אותה שם. אם המוסך מלא – הוא שם אותה בחנייה. הצרכן (המכונאי) לוקח רק מהמוסך.

**פתרון (עבור היצרן):**
*(הערה: במבחן בדרך כלל משתמשים בפונקציה `try_down` או בלוגיקה של `if`)*
```c
Producer:
1. <מייצר פריט>
2. if (A is not full) {
3.    down(emptyA);
4.    down(mutexA);
5.    PutInA();
6.    up(mutexA);
7.    up(fullA);
8. } else {
9.    down(emptyB);
10.   down(mutexB);
11.   PutInB();
12.   up(mutexB);
13.   up(fullB);
14. }
```

---

### תרגיל 6: קוראים-כותבים עם העדפת כותבים (רמת מבחן קשה)
**השאלה:** כותב שמגיע חייב לחסום קוראים **חדשים** מלהיכנס, כדי שהכותב לא יחכה לנצח.
**סמפור מפתח:** `BlockRead`.

**סיפור הדמיה: "השומר בכניסה לספרייה" 💂‍♂️**
הספרייה פתוחה. כשהמורה (הכותב) מגיע, הוא לא זורק את התלמידים שכבר בפנים, אבל הוא אומר לשומר בשער: "אל תכניס אף תלמיד חדש!". הוא מחכה שהספרייה תתרוקן, כותב, ורק אז אומר לשומר שאפשר להכניס שוב תלמידים.

**פתרון:**
*   **הכותב:** מבצע `down(BlockRead)` בשניה שהוא מגיע.
*   **הקורא:** השורה הראשונה שלו היא `down(BlockRead)` ואחריה מיד `up(BlockRead)`.
    *   אם הסמפור פנוי – הוא עובר מיד.
    *   אם הכותב "נעל" את השומר – הקורא נתקע בשורה 1.

---

## 3. חוקי הברזל לפתרון (Checklist למבחן)

1.  **חוק ה"סנדוויץ'":** כל שינוי של משתנה (`++` / `--`) חייב להיות עטוף ב-`down(mutex)` ו-`up(mutex)`.
2.  **חוק ה"סדר הקדוש":** תמיד בצע `down` על המשאב (כמות פריטים/מקומות) **לפני** ה-`down` על המיוטקס. הפוך = Deadlock.
3.  **חוק ה"ראשון והאחרון":** בבעיות קוראים-כותבים, רק הראשון שנכנס והאחרון שיוצא מבצעים פעולה על ה"צד השני".
4.  **חוק ה"שומר":** אם רוצים עדיפות, שמים סמפור בשורה הראשונה של התהליך המועדף פחות.

# סיכום קורס מערכות הפעלה (20594) - ניהול זיכרון וטבלאות דפים (שאלה 8)

כאן ריכזתי את כל הידע שצברתי על **שאלה 8**. זהו הנושא החישובי המרכזי במבחן. המטרה היא להבין איך המחשב מוצא נתונים בזיכרון ה-RAM בתוך עולם של כתובות וירטואליות.

## 1. מושגי יסוד - השפה של הזיכרון

כדי לפתור את השאלות, אני חייב לשלוט במושגים הבאים:

*   **זיכרון וירטואלי (Virtual Memory):** אשליה שנותנים לתוכנה שיש לה מרחב כתובות עצום ורציף.
*   **דף (Page):** יחידת החלוקה של הזיכרון הווירטואלי (ה"פרוסה" שהתוכנה רואה).
*   **מסגרת (Frame):** יחידת החלוקה של הזיכרון הפיזי (ה-RAM האמיתי). גודל המסגרת **תמיד** שווה לגודל הדף.
*   **היסט (Offset):** המיקום המדויק בתוך הדף. אם הדף הוא דף נייר, ה-Offset הוא מספר השורה. **הוא לעולם לא משתנה בתרגום מכתובת וירטואלית לפיזית.**
*   **טבלת דפים (Page Table):** המפה שאומרת: "דף מספר X של התוכנה נמצא כרגע במסגרת מספר Y ב-RAM".
*   **PTE (Page Table Entry):** שורה אחת בטבלה. היא מכילה את מספר המסגרת ודגלים (כמו ביט Valid).
*   **פסיקת דף (Page Fault):** מצב שבו התוכנה ביקשה דף שלא נמצא כרגע ב-RAM (אלא בדיסק).

### המתמטיקה של הזיכרון (חובה לשנן):
המחשב סופר בביטים (חזקות של 2). אלו המספרים שיופיעו תמיד:
*   $1KB = 2^{10} Bytes$ (10 bits)
*   $4KB = 2^{12} Bytes$ (12 bits) - **גודל הדף הכי נפוץ במבחנים**
*   $1MB = 2^{20} Bytes$ (20 bits)
*   $1GB = 2^{30} Bytes$ (30 bits)

---

## 2. מאגר תרגילים לתרגול (מרמת בסיס עד רמת מבחן)

### תרגיל 1: זיהוי גודל ה-Offset
**השאלה:** במערכת נתון שגודל דף הוא $8KB$. כמה ביטים בכתובת מוקצים עבור ה-**Offset**?
**סיפור הדמיה:** יש לך ארגז סוכריות שיכול להכיל 8,192 סוכריות. אתה רוצה לתת לכל סוכרייה מספר סידורי בבינארית. כמה ספרות (ביטים) אתה צריך בתווית כדי להגיע לכל סוכרייה?
**תשובה:**
1. נמיר לקילו-בתים: $8KB = 8 \times 1024 = 8,192$ בתים.
2. נמצא את החזקה של 2: $8,192 = 2^{13}$.
3. המעריך הוא מספר הביטים. **תשובה: 13 ביט.**

---

### תרגיל 2: פירוק כתובת בינארית
**השאלה:** נתונה כתובת וירטואלית בת 16 ביט: `101100 0000001010`. ידוע שגודל דף הוא $1KB$. מהו מספר הדף ומהו האופסט?
**סיפור הדמיה:** הכתובת היא כמו "מספר דירה" בבניין. החלק הימני הוא מספר הדירה בקומה, והחלק השמאלי הוא מספר הקומה. הקו המפריד נקבע לפי כמות הדירות בכל קומה.
**תשובה:**
1. גודל דף $1KB = 2^{10}$, לכן ה-Offset הוא **10 הביטים הימניים**.
2. נחתוך את הכתובת: `101100` (שמאל) | `0000001010` (ימין).
3. אופסט = `0000001010` (בעשרוני: 10).
4. מספר דף = `101100` (בעשרוני: 44).
**תשובה: דף 44, אופסט 10.**

---

### תרגיל 3: תרגום כתובת פשוט (עשרוני)
**השאלה:** נתונה כתובת וירטואלית **9,000**. גודל דף הוא $4KB$ ($4,096$ בתים). טבלת הדפים אומרת שדף 2 נמצא במסגרת 5. מהי הכתובת הפיזית?
**סיפור הדמיה:** הסטודנט מבקש: "אני צריך את שורה 9,000 במחברת שלי". הספרנית אומרת: "המחברת מחולקת לדפים של 4,096 שורות. זה אומר שאתה בדף השני שלך. הדף הזה נמצא כרגע על שולחן מספר 5 ב-RAM".
**תשובה:**
1. נמצא את מספר הדף: $9,000 / 4,096 = 2.19$ -> אנחנו ב**דף מספר 2**.
2. נמצא את האופסט (השארית): $9,000 \pmod{4096} = 808$.
3. תרגום פיזי: מסגרת 5 מתחילה בכתובת $5 \times 4,096 = 20,480$.
4. נוסיף את האופסט: $20,480 + 808 = 21,288$.
**תשובה: 21,288.**

---

### תרגיל 4: תכנון טבלה דו-שלבית (קלאסי למבחן)
**השאלה:** כתובת של 32 ביט. גודל דף $4KB$. גודל כל כניסה (PTE) הוא 4 בתים. תכנן חלוקה ל-2 רמות כך שכל טבלה תתפוס בדיוק דף אחד.
**סיפור הדמיה:** העיר כל כך גדולה שהאינדקס לא נכנס בדף אחד. אז יצרנו ספר אינדקסים: Level 1 אומר לך באיזה עמוד בספר לחפש, ו-Level 2 הוא העמוד עצמו שבו רשום מספר הבית.
**תשובה:**
1. **Offset:** גודל דף $4KB = 2^{12}$ -> **12 ביט**.
2. **כניסות בדף:** כמה שורות נכנסות בדף אחד? $4,096 / 4 = 1,024$ שורות.
3. **ביטים לאינדקס:** $1,024 = 2^{10}$ -> צריך **10 ביט** כדי לבחור שורה אחת.
4. **סיכום החלוקה:** 10 (Level 1) + 10 (Level 2) + 12 (Offset) = 32 ביט.
**תשובה: 10-10-12.**

---

### תרגיל 5: חישוב גודל טבלה בזיכרון
**השאלה:** תהליך משתמש ב-$1GB$ של זיכרון. גודל דף $4KB$, וכל PTE הוא 4 בתים. כמה זיכרון תופסת טבלת הדפים (ברמה אחת)?
**סיפור הדמיה:** אתה מנהל רשימת מלאי. לכל פריט במחסן יש שורה ברשימה. אם המחסן ענק, גם רשימת המלאי עצמה תתפוס המון מקום על המדף.
**תשובה:**
1. כמה דפים יש ב-1GB? $\text{1GB} / 4KB = 2^{30} / 2^{12} = 2^{18}$ דפים.
2. כל דף צריך שורה של 4 בתים: $2^{18} \times 4 \text{ bytes} = 2^{18} \times 2^2 = 2^{20} \text{ bytes}$.
3. נמיר ליחידות מוכרות: $2^{20} \text{ bytes} = \mathbf{1MB}$.
**תשובה: 1MB.**

---

### תרגיל 6: זיהוי Page Fault (רמת מבחן)
**השאלה:** נתונה טבלת דפים שבה עבור דף 3 רשום: `Frame: 0, Valid: 0`. גודל דף $4KB$. מהי הכתובת הפיזית עבור כתובת וירטואלית **13,000**?
**סיפור הדמיה:** אתה מגיע לספרנית ומבקש ספר. היא מסתכלת בכרטיסייה ורואה שרשום "לא נמצא" (Valid 0). זה לא משנה איזה מספר שורה ביקשת, הספר לא כאן.
**תשובה:**
1. נמצא את מספר הדף: $13,000 / 4,096 = 3.17$ -> אנחנו ב**דף מספר 3**.
2. נסתכל בטבלה: עבור דף 3 ה-Valid Bit הוא **0**.
3. **מסקנה:** הנתון לא נמצא ב-RAM.
**תשובה: פסיקת דף (Page Fault). אין כתובת פיזית כרגע.**

## 3. טיפים ודגשים אבסולוטיים (Checklist למבחן)

1.  **חוק שימור האופסט:** אם הכתובת הווירטואלית נגמרת ב-`0x123`, גם הכתובת הפיזית **חייבת** להיגמר ב-`0x123`.
2.  **גודל טבלה מקסימלי:** אם שואלים כמה דפים תופסת טבלה דו-שלבית עבור תהליך שמשתמש ב**כל** הזיכרון (בחלוקת 10-10-12):
    *   טבלה ראשית אחת (דף 1).
    *   עוד 1,024 טבלאות משניות (1,024 דפים).
    *   **סה"כ: 1,025 דפים.**
3.  **המרות מהירות:** 
    *   אם הכתובת היא 32 ביט, והאופסט הוא 12, נשארו 20 ביט לדפים.
    *   אם הכתובת היא 64 ביט, החישובים גדלים אבל הלוגיקה זהה.
4.  **ביט Valid:** הדבר הראשון שבודקים בתרגום. אם הוא 0 -> עצור הכל, רשום Page Fault.
