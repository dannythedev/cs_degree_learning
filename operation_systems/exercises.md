

# סיכום קורס מערכות הפעלה (20594) - סנכרון תהליכים (שאלה 7)

נושא הסנכרון עוסק בבעיה אחת פשוטה: מה קורה כשכמה תוכנות (תהליכים) מנסות לגעת באותו מידע בדיוק באותה שנייה? אם לא ננהל את זה נכון, המידע ייהרס או שהמחשב ייתקע לנצח.

## 1. המושגים והסיפור האחיד: "המסעדה העמוסה" 🍽️

כדי להבין סנכרון, נדמיין שמערכת ההפעלה היא מסעדה גדולה, והתהליכים במחשב הם האנשים במסעדה. כדי לשמור על סדר, יש לנו כלי שנקרא **סמפור (Semaphore)**. סמפור הוא קופסה שקופה עם כרטיסים.
*   **`down` (לקחת):** בן אדם ניגש לקופסה ורוצה לקחת כרטיס. אם יש כרטיס - הוא לוקח וממשיך מיד. **אם הקופסה ריקה - הוא נופל לישון ליד הקופסה** ולא זז עד שמישהו ישים שם כרטיס.
*   **`up` (לתת):** בן אדם שם כרטיס חדש בקופסה. אם מישהו ישן ליד הקופסה, הפעולה הזו **מעירה אותו מיד**.

**הכלים שלנו במסעדה:**
1.  **מיוטקס (Mutex) = "המפתח לשירותים" 🔑:** קופסה שיש בה רק **כרטיס אחד**. מי שלקח אותו (`down`), נעל את הדלת ואף אחד לא יכול להיכנס אחריו. כשסיים, הוא מחזיר אותו (`up`). משמש לשמירה על פעולות רגישות.
2.  **סמפור מניה (Counting) = "חניון המסעדה" 🅿️:** קופסה עם הרבה כרטיסים (למשל 10). סופרת כמה מקומות פנויים יש.
3.  **יצרן (Producer) וצרכן (Consumer) = "השף והמלצר":** השף מייצר מנות ושם על הדלפק. המלצר לוקח מנות מהדלפק.
4.  **קוראים וכותבים (Readers-Writers) = "הלקוחות ומנקה השטיחים":** לקוחות (קוראים) יכולים ללכת עשרות ביחד על השטיח. אבל כשהמנקה (הכותב) מגיע לנקות, הוא חייב שהשטיח יהיה **ריק לגמרי**.

---

## 2. חוקי הברזל לסנכרון במבחן

1.  **חוק ה"סנדוויץ'" (הגנה על מתמטיקה):** אם אתה רואה קוד שמעדכן מספר (כמו `count++`), חייבים לעשות עליו הגנה. לוקחים מפתח לשירותים לפני, ומחזירים אחרי:
    `down(mutex)` $\rightarrow$ `count++` $\rightarrow$ `up(mutex)`.
2.  **חוק "קודם חניה, ואז מפתח לשירותים":** למניעת קיפאון (Deadlock). אם תהליך צריך לעשות שני `down` רצופים, **תמיד** לוקחים קודם את כרטיס החניה (`empty` או `full`), ורק אז לוקחים את המפתח לשירותים (`mutex`). אם תעשה הפוך - תינעל בשירותים ותחכה שם לחניה. המערכת תיתקע לנצח.
3.  **חוק "הראשון מדליק אור, האחרון מכבה":** בקבוצה (כמו קוראים), לא כולם נועלים את הדלת. רק הראשון שנכנס מוודא שאין מנקה ונועל אותו בחוץ. רק האחרון שיוצא אומר למנקה: "סיימנו, אתה יכול להיכנס".

## 3. תרגול: מאפס לרמת מבחן

### תרגיל 1: המנעול הפשוט (Mutex)
**השאלה:** יש קופת טיפים משותפת (`Tips=0`). שני מלצרים מנסים לעשות `Tips++` בו-זמנית. השלם את הקוד. נתון: `mutex=1`.
**ההיגיון:** חוק הסנדוויץ'. אם שניהם יגעו בקופה יחד, מטבעות ייפלו. צריך מפתח לשירותים.
**הפתרון:**
```c
// Waiter:
1. down(mutex); // לוקח את המפתח (נועל את הקופה)
2. Tips++;      // מכניס טיפ בבטחה
3. up(mutex);   // מחזיר את המפתח
```

---

### תרגיל 2: תיאום בין שניים (סמפור מסנכרן)
**השאלה:** השף מכין סושי, המלצר מגיש. אסור למלצר להגיש כל עוד אין סושי מוכן. נתון סמפור `sushi = 0`.
**ההיגיון:** המלצר חייב "לישון" עד שהשף יכין וישים כרטיס בקופסה.
**הפתרון:**
```c
Chef (השף):
1. < מכין סושי >
2. up(sushi);   // נותן כרטיס "מוכן" (מעיר את המלצר)

Waiter (המלצר):
1. down(sushi); // מנסה לקחת כרטיס "מוכן". אם אין (0) - נרדם!
2. < מגיש ללקוח >
```

---

### תרגיל 3: יצרן-צרכן עם דלפק מוגבל (רמת מבחן קלה)
**השאלה:** הדלפק במסעדה יכול להכיל רק N מנות. השף נעצר אם מלא, המלצר נעצר אם ריק. 
נתונים: `empty=N` (כרטיסי מקום פנוי), `full=0` (כרטיסי מנות מוכנות), `mutex=1` (מפתח לשירותים למניעת התנגשות ידיים בדלפק).
**ההיגיון:** חוק "קודם חניה, ואז מפתח". קודם בודקים אם יש מקום/אוכל, ורק אז נועלים את הדלפק.
**הפתרון:**
```c
Chef (יצרן):
1. down(empty); // 1. מנסה לקחת כרטיס "מקום פנוי". 
2. down(mutex); // 2. לוקח מפתח לשירותים (נועל דלפק).
3. < מניח מנה >
4. up(mutex);   // 3. מחזיר מפתח.
5. up(full);    // 4. שם כרטיס "מנה מוכנה" למלצר.

Waiter (צרכן):
1. down(full);  // 1. מנסה לקחת כרטיס "מנה מוכנה".
2. down(mutex); // 2. לוקח מפתח.
3. < לוקח מנה >
4. up(mutex);   // 3. מחזיר מפתח.
5. up(empty);   // 4. שם כרטיס "התפנה מקום" לשף.
```

---

### תרגיל 4: מלכודת הקיפאון (Deadlock)
**השאלה:** מה יקרה אם השף בתרגיל 3 יחליף בין שורה 1 לשורה 2? (יעשה קודם `down(mutex)` ואז `down(empty)`).
**ההיגיון:** השף לקח את המפתח לשירותים, נעל את הדלפק. עכשיו הוא מסתכל בקופסת ה"מקום פנוי". אם אין מקום פנוי – הוא נופל לישון! 
המלצר בא, רוצה לקחת מנה כדי לפנות מקום, אבל הוא לא יכול לגשת לדלפק כי השף ישן עם המפתח בכיס! 
**תשובה:** קיפאון (Deadlock) מוחלט. אף אחד לא יזוז לעולם.

---

### תרגיל 5: ריבוי יצרנים (מפעל הממתקים ממבחן 68)
**השאלה:** במקום שף אחד, יש 10 שפים שמכינים מנות לאותו דלפק. מה צריך להוסיף לקוד של תרגיל 3?
**ההיגיון:** האם אנחנו צריכים מנעולים חדשים? לא! ה-`mutex` שכבר שמנו סביב הפעולה `< מניח מנה >` (חוק הסנדוויץ') מבטיח שגם אם 10 שפים יגיעו יחד, רק אחד מהם יחזיק את המפתח בכל רגע נתון.
**תשובה:** אין צורך לשנות כלום. המיוטקס מגן מפני התנגשויות בין יצרנים שונים.

---

### תרגיל 6: בחירה בין שני דלפקים (`Try_Down`)
**השאלה:** לשף יש דלפק A ודלפק B. הוא רוצה להניח מנה ב-A. אם A מלא, הוא יניח ב-B.
**ההיגיון:** השף לא יכול לעשות `down(empty_A)` רגיל, כי אם A מלא הוא יירדם שם ולא ילך ל-B! הוא צריך רק "להציץ" לקופסה.
**הפתרון:**
במבחנים משתמשים בפונקציה מיוחדת או פשוט בודקים ב-`if`.
```c
Chef:
if (empty_A > 0) {    // הצצה: האם יש מקום ב-A?
    down(empty_A);    // לוקח כרטיס בטוח
    down(mutex_A);
    < מניח ב-A >
    up(mutex_A);
    up(full_A);
} else {              // אם A מלא, נלך ל-B
    down(empty_B);    // כאן מותר לו כבר להירדם אם B מלא
    down(mutex_B);
    < מניח ב-B >
    up(mutex_B);
    up(full_B);
}
```

---

### תרגיל 7: קוראים וכותבים - הבסיס (הלקוחות מול המנקה)
**השאלה:** לקוחות (קוראים) יכולים להיות יחד על השטיח. מנקה (כותב) חייב להיות לבד על השטיח. נתון מונה `readers=0` וסמפור `ClearCarpet=1` שמגן על השטיח.
**ההיגיון:** חוק "הראשון והאחרון".
**הפתרון:**
```c
Customer (קורא):
1. down(mutex);         // סנדוויץ' סביב המונה
2. readers++;
3. if (readers == 1)    // אני הלקוח הראשון!
4.    down(ClearCarpet);// נועל את השטיח למנקה
5. up(mutex);

   < דורך על השטיח בשמחה עם כולם >

6. down(mutex);
7. readers--;
8. if (readers == 0)    // אני הלקוח האחרון שעוזב
9.    up(ClearCarpet);  // אומר למנקה: "כולם הלכו, תיכנס"
10. up(mutex);

Cleaner (כותב):
1. down(ClearCarpet);   // מחכה שכולם ילכו ונועל את השטיח
2. < מנקה את השטיח >
3. up(ClearCarpet);     // מסיים לנקות
```

---

### תרגיל 8: קוראים-כותבים עם עדיפות לכותבים (שאלת המבחן הקשה ממועד 72!) 👑
**השאלה:** במצב הרגיל (תרגיל 7), אם כל הזמן יבואו לקוחות חדשים, המנקה לעולם לא יצליח להיכנס (כי `readers` לא יגיע ל-0). אנחנו רוצים לתת לו **עדיפות**: כשהמנקה מגיע, אסור ללקוחות *חדשים* להיכנס.
**ההיגיון:** מוסיפים "שומר בשער המסעדה" (סמפור `BlockRead=1`). כשהמנקה מגיע, הדבר הראשון שהוא עושה זה לקחת לשומר את המפתח, כדי שיינעל את הדלת מבחוץ.
**הפתרון (לכותב / המנקה):**
```c
Cleaner (כותב עם עדיפות):
1. down(W_mutex);        // סנדוויץ' מונה מנקים
2. cleaners++;
3. if (cleaners == 1)    // המנקה הראשון שהגיע למסעדה!
4.    down(BlockRead);   // לוקח את המפתח מהשומר. לקוחות נחסמים בחוץ!
5. up(W_mutex);

6. down(ClearCarpet);    // מחכה שלקוחות שכבר בפנים יסיימו ויצאו
7. < מנקה את השטיח >
8. up(ClearCarpet);

9. down(W_mutex);
10. cleaners--;
11. if (cleaners == 0)   // המנקה האחרון שיוצא
12.    up(BlockRead);    // מחזיר לשומר את המפתח. הלקוחות מוזמנים להיכנס.
13. up(W_mutex);
```
*(הקורא רק יעשה `down(BlockRead)` ואז `up(BlockRead)` בשורה 1 שלו, רק כדי "לעבור את השומר" ולוודא שהשער פתוח).*

### תרגיל 9: למה צריך את `W_mutex` אצל הכותבים בתרגיל 8?
**השאלה:** למה צריך להגן על `cleaners++` עם מיוטקס משלו? למה פשוט לא לכתוב את זה ישירות?
**ההיגיון:** חוק הסנדוויץ'. פעולת הפלוס-פלוס (`++`) היא לא פעולה "אטומית" במעבד. המחשב קורא את המספר, מוסיף 1, וכותב חזרה. אם שני מנקים יגיעו בדיוק באותה המילי-שנייה, הם עלולים לקרוא "0", שניהם יוסיפו 1, ושניהם יחשבו שהם המנקה הראשון. שניהם ינסו לנעול את השומר וייווצר קלקול.
**תשובה:** חובה להשתמש במיוטקס כדי למנוע **מרוץ קריטי (Race Condition)** על המשתנה שסופר את הכותבים.

---

### תרגיל 10: סנכרון עם מגבלת משאבים אקראית
**השאלה:** לקוח רוצה לקנות בירה. במקרר יש `Beers=50`. איך הלקוח מוציא בירה בלי להרוס את הכמות?
**הפתרון:** זו שאלת סיכום פשוטה. אם אין לנו מחסן עם מקומות, אלא רק משתנה שמייצג כמות, פשוט מגינים עליו!
```c
1. down(mutex);  // תופס את מפתח המקרר
2. if (Beers > 0) Beers--; // לוקח בירה אם יש
3. up(mutex);    // משחרר את המקרר
```

# סיכום קורס מערכות הפעלה (20594) - ניהול זיכרון וטבלאות דפים (שאלה 8)

כדי לפתור כל שאלת זיכרון במבחן, נשתמש באנלוגיית **"בנייני המשרדים"**.
*   **מרחב וירטואלי:** עיר דמיונית ענקית שהלקוח (התוכנה) חושב ששייכת רק לו.
*   **דף (Page):** בניין משרדים בעיר הדמיונית.
*   **מסגרת (Frame):** חלקה פנויה בעיר האמיתית (זיכרון ה-RAM הפיזי). **חוק טבע:** גודל חלקה תמיד שווה בדיוק לגודל בניין!
*   **היסט (Offset):** מספר החדר בתוך הבניין. **חוק ברזל:** מספר החדר לעולם לא משתנה! חדר 10 בבניין הדמיוני, יהיה חדר 10 גם כשנניח את הבניין על חלקה אמיתית.
*   **טבלת דפים (Page Table):** ספר הרישום של פקיד העירייה, שמתרגם: "הבניין הדמיוני מספר X מונח כרגע בחלקה האמיתית מספר Y".
*   **PTE (Page Table Entry):** שורה אחת בתוך ספר הרישום.

### הכלל המתמטי היחיד שצריך:
במחשבים אין 10 ספרות, יש רק 0 ו-1 (ביטים). כדי לדעת כמה ביטים (ספרות בינאריות) צריך כדי למספר משהו, מחפשים את החזקה של 2.
*   $2^{10}$ אפשרויות = $1,024$ (נקרא **1KB**)
*   $2^{12}$ אפשרויות = $4,096$ (נקרא **4KB**)
*   $2^{20}$ אפשרויות = מיליון (נקרא **1MB**)
*   $2^{30}$ אפשרויות = מיליארד (נקרא **1GB**)

## תרגיל 1: מציאת ה-Offset (כמה ספרות על הדלת?)
**השאלה:** נתונה מערכת עם דפים בגודל $8KB$. כמה ביטים בכתובת מוקצים עבור ה-Offset?
**ההיגיון (הסיפור):** קיבלת בניין עם 8,192 חדרים ($8 \times 1024$). אתה צריך להדפיס תווית עם מספר בינארי לכל דלת בבניין. כמה ספרות (ביטים) תצטרך על התווית כדי שתוכל לייצר 8,192 קומבינציות שונות (מחדר 0 עד 8191)?
**הפתרון:**
1. נמיר את גודל הדף (הבניין) לבתים (חדרים): $8KB = 8 \times 1024 = 8,192$ חדרים.
2. נמצא איזו חזקה של 2 נותנת 8,192. 
3. מכיוון ש-$2^{13} = 8192$, אנחנו חייבים לשריין 13 ספרות בינאריות לכל תווית.
**תשובה: 13 ביט מוקצים ל-Offset.**

---

## תרגיל 2: חיתוך הכתובת (קריאת הפתק)
**השאלה:** במערכת עם כתובת של 16 ביט וגודל דף של $1KB$, לקוח מוסר את הכתובת הבינארית: `101111 0000000101`. מהו מספר הדף (Page) ומהו האופסט (Offset)?
**ההיגיון (הסיפור):** הלקוח הביא לך פתק עם 16 ספרות. אתה יודע שהספרות הימניות הן תמיד "מספר החדר", והשמאליות הן "מספר הבניין". איפה אתה מעביר את קו החיתוך בעט?
**הפתרון:**
1. נבדוק כמה חדרים יש בבניין: $1KB = 1,024$ חדרים. 
2. כדי למספר 1024 חדרים, צריך 10 ספרות ($2^{10}$). כלומר, ה-Offset תופס את **10 הביטים הימניים**.
3. נחתוך את הפתק: `101111` (שמאל - בניין) | `0000000101` (ימין - חדר).
4. מספר הבניין (Page): `101111` (בבינארית זה 47).
5. מספר החדר (Offset): `0000000101` (בבינארית זה 5).
**תשובה: דף 47, אופסט 5.**

---

## תרגיל 3: התרגום בפועל (מעבר לעיר האמיתית)
**השאלה:** הכתובת המבוקשת היא **9,000**. גודל דף הוא $4,096$ בתים. בספר הרישום כתוב שדף 2 נמצא במסגרת 5. מהי הכתובת הפיזית האמיתית ב-RAM?
**ההיגיון (הסיפור):** הלקוח מחפש את חדר 9,000 בעיר הדמיונית. הפקיד צריך לחשב באיזה בניין נמצא החדר הזה, למצוא איפה הבניין הזה הונח בעיר האמיתית, ולכוון את הלקוח לחדר המדויק.
**הפתרון:**
1. **באיזה בניין (Page) אנחנו?** נחלק את הכתובת בגודל הבניין: $9000 / 4096 = 2.19$. הלקוח נמצא ב**בניין מספר 2**.
2. **איזה חדר בתוך הבניין (Offset)?** השארית של החילוק. $9000 - (2 \times 4096) = 9000 - 8192 = \mathbf{808}$. הלקוח צריך את **חדר 808**. (החדר הזה לא ישתנה לעולם!).
3. **איפה הבניין באמת?** הפקיד מסתכל בספר הרישום: בניין דמיוני 2 הונח על **חלקה 5** בעיר האמיתית.
4. **חישוב הכתובת האמיתית:** חלקה 5 בעיר האמיתית מתחילה בכתובת $5 \times 4096 = 20,480$.
5. נוסיף את חדר 808 לכתובת ההתחלה: $20,480 + 808 = \mathbf{21,288}$.
**תשובה: הכתובת הפיזית היא 21,288.**

---

## תרגיל 4: זיהוי Page Fault (המלכודת במבחן)
**השאלה:** גודל דף $4KB$. כתובת וירטואלית מבוקשת: 15,000. בספר הרישום של הפקיד, בשורה של דף 3 רשום: `Frame = 10, Valid = 0`. מה תהיה הכתובת הפיזית?
**ההיגיון (הסיפור):** הלקוח מחפש את חדר 15,000. נחשב באיזה בניין זה: $15000 / 4096 = 3.66$ (בניין 3). הפקיד פותח את הספר בשורה של בניין 3 ורואה שיש שם מדבקה אדומה: `Valid = 0`. זה אומר שהבניין הזה בכלל לא נמצא בעיר (ה-RAM) כרגע! הוא באחסון מחוץ לעיר (בדיסק הקשיח).
**הפתרון:** אם Valid שווה 0, אין כתובת פיזית! הפקיד צריך לעצור הכל ולהביא את הבניין.
**תשובה: תתרחש פסיקת דף (Page Fault).**

---

## תרגיל 5: למה מסבכים אותנו? (הצורך בטבלאות מרובות רמות)
**השאלה:** במערכת של 32 ביט, עם דפים של $4KB$, ושורה בספר (PTE) ששוקלת 4 בתים. כמה מקום יתפוס "ספר רישום" רגיל ושטוח בזיכרון ה-RAM עבור תוכנה אחת?
**ההיגיון (הסיפור):** כתובת של 32 ביט אומרת שללקוח יש עיר ענקית עם $2^{32}$ חדרים! אם נחלק אותה לבניינים של 4,096 חדרים, כמה שורות פקיד העירייה יצטרך לכתוב בספר שלו?
**הפתרון:**
1. כמות החדרים בעיר: $2^{32}$ (4 מיליארד).
2. חדרים בכל בניין: $2^{12}$ ($4,096$).
3. כמות הבניינים בעיר: $2^{32} / 2^{12} = \mathbf{2^{20}}$ בניינים (מיליון בניינים!).
4. הפקיד צריך לרשום 1,000,000 שורות. כל שורה שוקלת 4 בתים.
5. משקל הספר: $2^{20} \times 4 \text{ bytes} = 4 \text{ MB}$.
**המסקנה:** 4 מגה-בייט רק בשביל ספר רישום אחד זה מטורף! לכן המציאו את התרגיל הבא.

---

## תרגיל 6: תכנון טבלה רב-שלבית (שאלת המבחן הקבועה!) 👑
**השאלה:** כדי לחסוך מקום, החליטו לחלק את הספר הגדול לספרי אינדקס קטנים (טבלה רב-שלבית). **הדרישה:** כל ספר אינדקס קטן חייב להיכנס בדיוק בתוך **דף אחד** (בניין אחד). 
כתובת 32 ביט, גודל דף $4KB$, שורה שוקלת 4 בתים. תכנן חלוקה ל: `רמה 1`, `רמה 2`, `Offset`.
**ההיגיון (הסיפור):** אנחנו צריכים שספר האינדקס ייכנס בדיוק לתוך בניין של 4,096 חדרים. נחשב כמה שורות אפשר להכניס בבניין אחד, ולפי זה נקבע את מספר הספרות בתווית.
**הפתרון:**
1. **האופסט (תמיד קבוע!):** בניין של 4KB דורש **12 ביט** לחדרים ($2^{12}$).
2. **כמה שורות נכנסות לספר (בניין) אחד?**
   הבניין הוא בגודל 4,096 בתים. כל שורה צריכה 4 בתים.
   $4096 / 4 = \mathbf{1024}$ שורות נכנסות בספר אחד.
3. **כמה ביטים צריך כדי למספר 1,024 שורות?** 
   $1024 = 2^{10}$. לכן, צריך **10 ביט** לאינדקס.
4. **הסיכום:** 
   רמה 1 (האינדקס הראשי) תקבל 10 ביט.
   רמה 2 (האינדקס המשני) תקבל 10 ביט.
   האופסט (חדרים) מקבל 12 ביט.
   *(בדיקה: 10 + 10 + 12 = 32. מושלם!)*.
**תשובה: 10-10-12.**

---

## תרגיל 7: חלוקה עם נתונים הפוכים
**השאלה:** נתונה מערכת של 32 ביט. טבלה של שתי רמות. נתון שרמה 1 דורשת **8 ביט**, ורמה 2 דורשת **8 ביט**. מהו גודל הדף (הבניין) במערכת הזו?
**ההיגיון (הסיפור):** הפעם אנחנו יודעים כמה ספרות לוקח למספר את הבניינים ($8+8=16$). אנחנו נבדוק כמה ספרות נשארו לחדרים (לאופסט), ולפי זה נגלה כמה חדרים יש בבניין!
**הפתרון:**
1. הכתובת הכוללת היא 32 ביט.
2. הופחתו לאינדקסים: $8 + 8 = 16$ ביט.
3. נשאר לאופסט: $32 - 16 = \mathbf{16}$ ביט.
4. אם האופסט הוא 16 ביט, כמות החדרים בבניין היא $2^{16}$ בתים.
5. $2^{16} = 2^6 \times 2^{10} = 64 \times 1KB = \mathbf{64KB}$.
**תשובה: גודל הדף הוא 64KB.**

---

## תרגיל 8: משקל הטבלאות בזיכרון הפיזי
**השאלה:** בחלוקה של 10-10-12. אם התוכנה משתמשת ב**כל** הזיכרון הווירטואלי שלה, כמה דפים (ארגזים ב-RAM) יתפסו **כל** ספרי האינדקס יחד (ראשי + כל המשניים)?
**ההיגיון (הסיפור):** תוכנה שמשתמשת בכל העיר צריכה שנכין לה את כל ספרי הרישום האפשריים. 
*   יש ספר ראשי אחד. הוא מכיל 1,024 שורות (כי יש לו 10 ביט). 
*   כל שורה בספר הראשי מפנה לספר משני שלם! אז נצטרך להדפיס 1,024 ספרים משניים.
**הפתרון:**
1. ספר ראשי אחד = 1 דף.
2. 1,024 ספרים משניים = 1,024 דפים.
**תשובה: סך הכל 1,025 דפים יתפסו רק לצורך ניהול הטבלאות.**

---

## תרגיל 9: היתרון של רב-שלבי (תוכנית קטנה)
**השאלה:** באותה מערכת של 10-10-12, מה יקרה אם התוכנה קטנטנה ומשתמשת רק ב-2 דפים? כמה דפים יתפסו הטבלאות שלה?
**ההיגיון (הסיפור):** כשהלקוח קטן ושוכר רק 2 בניינים, אין טעם להדפיס לו 1,024 ספרי אינדקס משניים שהוא לא ישתמש בהם!
**הפתרון:**
1. תמיד נדפיס **ספר ראשי אחד** (1 דף).
2. שני הבניינים שהוא שכר נמצאים כנראה קרובים זה לזה. לכן נצטרך להדפיס רק **ספר משני אחד** (1 דף) כדי לנהל את שניהם.
**תשובה: הטבלאות יתפסו רק 2 דפים.** (לעומת 1,025 בתרגיל הקודם. זה החיסכון האדיר של שיטת הרב-שלבי!).

---

## תרגיל 10: "הפתקית של הפקיד" (TLB)
**השאלה:** מה תפקידו של ה-TLB (Translation Lookaside Buffer)?
**ההיגיון (הסיפור):** בכל פעם שלקוח בא לפקיד, הפקיד קם, הולך לספר הראשי, ואז הולך לספר המשני. זה לוקח זמן. אבל אם הלקוח מבקש את אותו בניין 5 פעמים ברצף? הפקיד לא טיפש. אחרי הפעם הראשונה, הוא רושם לעצמו על פתק צהוב קטן ליד המקלדת: "בניין דמיוני 2 = חלקה אמיתית 5".
**הפתרון:**
ה-TLB הוא זיכרון מטמון (Cache) קטנצ'יק וסופר-מהיר בתוך המעבד. הוא שומר את תרגומי הכתובות האחרונים שנעשו. לפני שהמעבד הולך לטבלאות הכבדות ב-RAM, הוא מציץ ב-TLB. אם התרגום שם (TLB Hit), חסכנו המון זמן.

---


# סיכום קורס מערכות הפעלה (20594) - מערכת קבצים ו-I-node (שאלה 9)

הנושא הזה עוסק בשאלה אחת: איך המחשב שומר קבצים ענקיים על הדיסק הקשיח, ואיך הוא מוצא את המידע בתוכם במהירות?

## 1. המושגים והסיפור האחיד: "מחסן הארכיון הענק" 📦

דמיין ש**קובץ** במחשב הוא בעצם **אנציקלופדיה בת עשרות אלפי עמודים**.
אנחנו מנהלים מחסן ארכיון ענק, ויש לנו חוקים ברורים לאחסון:

1.  **בלוק (Block) = "ארגז אחסון":** אי אפשר לשים דפים בודדים במחסן. אנחנו אורזים הכל בתוך ארגזי קרטון. כל הארגזים בעולם הם **בדיוק באותו גודל** (למשל, כל ארגז מכיל בדיוק 4,096 תווים של טקסט - 4KB). לכל ארגז במחסן יש מספר סידורי.
2.  **מצביע (Pointer / Address) = "פתקית ניווט":** פתקית קטנה שכתוב עליה מספר של ארגז (למשל "ארגז מס' 805"). הפתקית הזו שוקלת מעט מאוד (בדרך כלל 4 או 8 בתים).
3.  **I-node = "כרטיסיית הקטלוג":** זוהי תעודת הזהות של האנציקלופדיה. היא נמצאת אצל מנהל המחסן (בזיכרון), ובה רשומים פרטים על הקובץ, ובעיקר - **מודבקות עליה פתקיות ניווט**.
4.  **מצביעים ישירים (Direct Pointers):** ה-10 (או 12) הפתקיות הראשונות שמודבקות על כרטיסיית ה-I-node. הן אומרות לך: "הנה 10 הארגזים הראשונים של האנציקלופדיה".
5.  **מצביע עקיף יחיד (Single Indirect):** הפתקית ה-11 על הכרטיסייה. היא שולחת אותך לארגז במחסן, אבל הפתעה! הארגז הזה לא מכיל דפים של האנציקלופדיה. הוא **"ארגז קלסרים" שמלא כולו בפתקיות ניווט נוספות!**
6.  **מצביע עקיף כפול (Double Indirect):** הפתקית ה-12 על הכרטיסייה. שולחת אותך לארגז, שמלא בפתקיות, ששולחות אותך לארגזים אחרים שמלאים בפתקיות. (ארגז של ארגזי פתקיות).

## 2. חוקי הברזל למתמטיקה של הקבצים

כדי לפתור כל שאלה, צריך לחשב את **חוק ה-P (כמה פתקיות נכנסות בארגז?)**:
*   מכיוון שהארגזים אצלנו הם בגודל קבוע, השאלה הראשונה היא תמיד: "אם אני ממלא ארגז שלם רק בפתקיות ניווט, כמה פתקיות ייכנסו בו?".
*   **הנוסחה:** `P = (גודל הבלוק / גודל המצביע)`. 
*   **לדוגמה:** אם גודל בלוק הוא 4,096 בתים, ופתקית שוקלת 8 בתים, אז $4096 / 8 = \mathbf{512}$ פתקיות בארגז.

---

## 3. תרגול: מאפס לרמת מבחן

### תרגיל 1: חישוב ה-P (הבסיס של הבסיס)
**השאלה:** נתון מערכת קבצים שבה גודל בלוק הוא $2KB$. אורך כל מצביע (כתובת בלוק בדיסק) הוא 4 בתים. כמה מצביעים ייכנסו בבלוק עקיף אחד?
**ההיגיון:** יש לי ארגז בנפח 2,048 בתים. כל פתקית שוקלת 4 בתים. נחלק כדי לראות כמה ייכנסו.
**הפתרון:**
1. $2KB = 2 \times 1024 = 2,048$ בתים.
2. נחלק: $2048 / 4 = 512$.
**תשובה: 512 מצביעים ($P = 512$).**

---

### תרגיל 2: גודל קובץ מינימלי (רק ארגזי טקסט)
**השאלה:** ב-I-node יש 10 מצביעים ישירים. גודל בלוק $4KB$. מהו גודל הקובץ המקסימלי שניתן לשמור מבלי להשתמש במצביעים עקיפים בכלל?
**ההיגיון:** מנהל המחסן נותן לך לנצל רק את 10 הפתקיות שמודבקות לו על הכרטיסייה. כמה טקסט תוכל לאחסן?
**הפתרון:**
1. 10 פתקיות ישירות = 10 ארגזים מלאים בטקסט.
2. $10 \times 4KB = 40KB$.
**תשובה: 40KB.**

---

### תרגיל 3: הוספת ארגז פתקיות (עקיף יחיד)
**השאלה:** המשך לתרגיל הקודם ($4KB$ בלוק). נתון שבבלוק עקיף נכנסים $512$ מצביעים. אם הקובץ מנצל את כל המצביעים הישירים ואת המצביע העקיף היחיד בשלמותו, כמה מידע (data) יש בקובץ?
**ההיגיון:** הלקוח מילא את 10 הארגזים הראשונים. עכשיו הוא פתח "ארגז פתקיות" אחד שמאפשר לו להגיע לעוד 512 ארגזים חדשים של טקסט!
**הפתרון:**
1. מידע מ-10 ישירים: $10 \times 4KB = 40KB$.
2. מידע מהעקיף היחיד: 512 פתקיות $\times$ ארגז של $4KB$.
   $512 \times 4KB = 2,048KB$ (שזה 2 מגה-בייט).
3. נסכום את שניהם: $2048 + 40 = 2088KB$.
**תשובה: 2,088KB.**

---

### תרגיל 4: המפלצת (עקיף כפול)
**השאלה:** המשך לתרגיל 3. הלקוח ממלא גם את המצביע העקיף הכפול במלואו. כמה מידע זה מוסיף?
**ההיגיון:** הפתק ה-12 בכרטיסייה מפנה ל"ארגז פתקיות ראשי". בתוכו יש 512 פתקיות שמפנות כל אחת ל"ארגז פתקיות משני". בכל ארגז משני יש 512 פתקיות שמפנות סוף סוף לארגזי טקסט אמיתיים!
**הפתרון:**
1. כמות ארגזי הטקסט בסוף השרשרת: $512 \times 512 = 262,144$ ארגזים!
2. נכפיל בגודל ארגז: $262,144 \times 4KB = 1,048,576KB$.
3. נמיר ליחידות נורמליות (נחלק ב-1024): זה שווה ל-$1,024MB$, שזה בדיוק **1GB**.
**תשובה: העקיף הכפול לבדו מאפשר להוסיף 1GB של נתונים לקובץ!**

---

### תרגיל 5: למצוא את המחט בערימת השחת (אינדקס לוגי)
**השאלה:** גודל בלוק $1KB$. הלקוח מבקש לקרוא את הבייט ה-**3,500** בתוך הקובץ שלו. באיזה בלוק "לוגי" (מ-0 והלאה) של הקובץ נמצא הבייט הזה?
**ההיגיון:** קראת את האנציקלופדיה שלך, ואתה מחפש את התו (האות) ה-3,500 מתחילת הסיפור. באיזה "ארגז טקסט" היא נמצאת, אם כל ארגז מכיל 1,024 אותיות?
**הפתרון:**
1. נחלק: $3,500 / 1024 = 3.41$.
2. זה אומר שמילאת את ארגזים 0, 1, ו-2 לגמרי. עכשיו התחלת את הארגז הרביעי (שהאינדקס שלו הוא 3).
**תשובה: בלוק לוגי מספר 3.**

---

### תרגיל 6: מאיפה יוצאים? (ניתוב ב-I-node)
**השאלה:** חישבת שהבייט שהלקוח מחפש נמצא ב**בלוק הלוגי מספר 500**. ב-I-node יש 10 ישירים. בבלוק עקיף נכנסים 256 מצביעים ($P=256$). מאיזה מצביע ב-I-node תתחיל את החיפוש?
**ההיגיון:** אנחנו מחפשים את ארגז הטקסט ה-500. נבדוק לאיזו "קטגוריה" בכרטיסייה הוא שייך.
**הפתרון:**
1. **ארגזים 0 עד 9:** שייכים ל-10 המצביעים הישירים. (אנחנו מחפשים 500, אז זה לא פה).
2. **ארגזים 10 עד 265:** שייכים לעקיף יחיד (כי הוא מכיל 256 פתקיות, והתחלנו לספור מ-10. $10+256-1 = 265$). (זה לא פה).
3. **ארגזים 266 והלאה:** שייכים לעקיף כפול!
**תשובה: החיפוש יעבור דרך המצביע העקיף הכפול.**

---

### תרגיל 7: כמה פעמים נסענו למחסן? (Disk Access)
**השאלה:** המשך לתרגיל 6. בהנחה שאין Cache (מטמון), כמה "גישות לדיסק" (נסיעות למחסן) ידרשו כדי לקרוא את ארגז הטקסט ה-500? ה-I-node כבר אצלנו ביד.
**ההיגיון:** אנחנו בנתיב של העקיף הכפול. כל פעם שאתה מרים ארגז כדי לחפש בתוכו פתקית – זו נסיעה למחסן הדיסק.
**הפתרון:**
1. נסיעה ראשונה: לקרוא את ארגז הפתקיות הראשי (עקיף כפול).
2. נסיעה שנייה: לקרוא את ארגז הפתקיות המשני שמצאנו.
3. נסיעה שלישית: לקרוא סוף סוף את ארגז הטקסט האמיתי שהפתקית הפנתה אליו.
**תשובה: 3 גישות לדיסק.**

---

### תרגיל 8: למה לא הכל ישיר? (שאלת הבנה למבחן)
**השאלה:** למה מתכנני לינוקס יצרו את הבלגן של המצביעים העקיפים? למה לא לשים מראש מיליון מצביעים ישירים בתוך ה-I-node וזהו?
**ההיגיון:** למה לא לעשות לכל אדם שמבקש לפתוח "תיקייה" כרטיסיית קטלוג בגודל של שולחן אוכל?
**הפתרון:** 
רוב הקבצים במחשב (טקסט, סקריפטים, קבצי הגדרות) הם קטנים מאוד (פחות מ-10 בלוקים). אם ה-I-node של כל קובץ היה ענק מראש, היינו מבזבזים כמויות אדירות של מקום בזיכרון על פתקיות ריקות. שיטת העקיפים מאפשרת ל-I-node להיות קטנצ'יק, ורק אם הקובץ הופך למפלצת – הוא "שוכר" ארגזי פתקיות נוספים.

---

### תרגיל 9: HDD מול SSD (הקלאסיקה של האמריקאיות)
**השאלה:** בדיסק קשיח ישן (HDD), קריאת קובץ ששמור ברצף (בלוקים 100,101,102) מהירה בהרבה מקריאת קובץ ששמור באקראי (בלוקים 100, 900, 20). האם זה נכון גם בכונן SSD חדיש?
**ההיגיון:** בדיסק ישן יש **זרוע מכנית** (כמו פטיפון של תקליטים). לנסוע מ-100 ל-900 לוקח זמן פיזי! ב-SSD הכל זה צ'יפים אלקטרוניים, אין שום דבר שזז.
**הפתרון:**
ב-SSD בעיית ה"מרחק הפיזי" (Seek Time) כמעט נעלמת. לכן ההבדל במהירות בין קובץ רציף לקובץ מקוטע קטן משמעותית לעומת HDD. (עם זאת, יש לציין שעדיין יש עניין קל של לוקאליות, אבל ההבדל תהומי לעומת HDD).

---

### תרגיל 10: תרגיל "השעווה" של המבחן - מידע מול ניהול 👑
**(מבוסס על שאלת מבחן אמיתית קשוחה):** 
**השאלה:** מערכת עם $P=512$ פתקיות בארגז. גודל בלוק $2KB$. נתון שהקובץ תפס **1,514 בלוקים סך הכל בדיסק** (המספר 1514 כולל גם את ארגזי הטקסט וגם את ארגזי הפתקיות/ניהול!). 
כמה מתוך ה-1514 הם ארגזי מידע (Data Blocks) אמיתיים? 
*(ה-Inode לא נספר ב-1514).*

**ההיגיון:** מנהל המחסן אומר לך: "הבאתי לך 1514 ארגזים". אתה צריך למיין אותם: כמה מהם מלאים בטקסט, וכמה מהם זה רק ארגזי פתקיות? נתחיל למלא את המבנה לפי הסדר!

**הפתרון השלבי (לנשום עמוק):**
1. **הישירים:** 10 הארגזים הראשונים הם טקסט נטו. (לא בזבזנו פה ארגזי ניהול).
   * מנוצל: 10 טקסט. נשארו לחלוקה: 1,504 ארגזים.
2. **העקיף היחיד:** פותחים אותו! זה אומר שלקחנו **1 ארגז ניהול**. הארגז הזה יכול להכיל 512 ארגזי טקסט. האם נמלא אותו? כן, כי יש לנו עוד 1500 ארגזים לחלק!
   * מנוצל בשלב זה: 1 ניהול + 512 טקסט = 513 ארגזים.
   * סך הכל ניצלנו עד כה: 10 + 513 = 523 ארגזים.
   * נשארו לחלוקה: $1,514 - 523 = \mathbf{991}$ ארגזים!
3. **העקיף הכפול:** עכשיו אנחנו עוברים אליו. קודם כל נשלוף **1 ארגז ניהול ראשי**.
   * נשארו לנו 990 ארגזים לחלק.
   * הארגז הראשי מצביע על ארגזי ניהול משניים. כל משני כזה מכיל 512 טקסט.
   * נפתח ארגז ניהול משני ראשון: ניקח 1 ניהול + 512 טקסט = 513 ארגזים.
   * נשארו: $990 - 513 = \mathbf{477}$ ארגזים לסיום!
   * האם צריך עוד ארגז משני? כן. נפתח ארגז משני נוסף (1 ניהול). נשארו 476.
   * את כל ה-476 נשים כארגזי טקסט תחתיו.
4. **בוא נספור רק את המידע (טקסט):**
   * מהישירים: 10.
   * מהעקיף יחיד: 512.
   * מהעקיף כפול משני א': 512.
   * מהעקיף כפול משני ב': 476.
   * **סה"כ מידע:** $10 + 512 + 512 + 476 = \mathbf{1,510}$ ארגזים של מידע.
   *(בדיקה עצמית לניהול: 1 עקיף יחיד + 1 כפול ראשי + 2 כפול משניים = 4 ארגזי ניהול. 1510+4 = 1514. בול!)*.

**תשובה: יש 1,510 בלוקים של מידע בקובץ.**

# סיכום קורס מערכות הפעלה (20594) - קיפאון ואלגוריתם הבנקאי

נושא זה עוסק בשאלה: מה קורה כשכמה תוכנות (תהליכים) "רבות" על משאבים במחשב (כמו מדפסת, זיכרון, קבצים), ואיך מוודאים שהן לא ייתקעו לנצח במאבק הזה.

## 1. המושגים והסיפור האחיד: "מנהל הבנק והקבלנים" 🏦🛠️

נחלק את הנושא לשניים: **הבעיה** (קיפאון) ו**הפתרון** (אלגוריתם הבנקאי).

### החלק הראשון: הבעיה - קיפאון (Deadlock)
**הסיפור: "שני הפועלים העקשנים" 👷‍♂️**
פועל א' ופועל ב' צריכים לבנות ארון. כל אחד חייב גם פטיש וגם מסמר כדי לסיים.
*   פועל א' תפס את הפטיש.
*   פועל ב' תפס את המסמר.
עכשיו פועל א' מבקש את המסמר ומחכה ש-ב' יסיים. פועל ב' מבקש את הפטיש ומחכה ש-א' יסיים. 
אף אחד לא מוכן לוותר על מה שיש לו ביד. הם יחכו ככה לנצח. **זהו קיפאון (Deadlock)**.

כדי שייווצר קיפאון חייבים להתקיים **4 תנאים יחד** (אם נשבור אחד, הקיפאון מתבטל!):
1.  **מניעה הדדית:** אי אפשר לחלוק את הכלי. רק אחד יכול להחזיק פטיש.
2.  **החזקה והמתנה:** הפועל מחזיק כלי אחד, ובמקביל מחכה לכלי שני.
3.  **אין הפקעה:** אסור למנהל העבודה לבוא ולקחת לפועל את הפטיש מהיד בכוח.
4.  **המתנה מעגלית:** א' מחכה ל-ב', שמחכה ל-ג', שמחכה ל-א'. (מעגל סגור של עקשנים).

### החלק השני: הפתרון - אלגוריתם הבנקאי 💰
כדי למנוע את הקיפאון מראש, המחשב מתנהג כמו **מנהל בנק זהיר מאוד**. 
הוא מנהל קבלנים (תהליכים) שלוקחים ממנו הלוואות (משאבים) כדי לסיים פרויקטים. 

*   **קופה זמינה (`Available`):** כמה כסף/משאבים נשארו כרגע פנויים לבנקאי בכספת.
*   **הקצאה (`Allocation`):** כמה כסף כבר נתנו לקבלן עד עכשיו (מה שיש לו ביד).
*   **דרישת מקסימום (`Max`):** ההצהרה של הקבלן ביום הראשון: "כדי לסיים את כל הפרויקט, אני אצטרך במקסימום X שקלים". (אם לא יקבל הכל, הוא לא יסיים את הפרויקט, ולא יחזיר את ההלוואה!).
*   **צורך (`Need`):** כמה כסף הקבלן **עוד צריך** לבקש כדי לסיים את הפרויקט. (`Need = Max - Allocation`).
*   **מצב בטוח (`Safe State`):** מצב שבו הבנקאי אומר: "עם הכסף שנשאר לי עכשיו בקופה, אני יכול להציל קבלן אחד שייקח את מה שחסר לו, יסיים, ויחזיר לי את הכל. עם הכסף שלו אציל את השני, וכך הלאה עד שכולם יסיימו".

---

## 2. חוקי הברזל למבחן

1.  **משוואת הזהב:** תמיד מתחילים מלחשב את הצורך של כולם: `Need = Max - Allocation`.
2.  **החוק של הבנקאי:** מנהל הבנק יאשר לקבלן להתקדם רק אם הצורך הנוכחי שלו (`Need`) קטן או שווה למה שיש כרגע בקופה (`Available`).
3.  **חוק ההחזרה:** ברגע שקבלן "סיים" (בסימולציה שלנו), הוא **מחזיר לקופה את כל מה שהיה לו ביד (`Allocation`)**. הקופה גדלה!

---

## 3. תרגול: מאפס לרמת מבחן

### תרגיל 1: זיהוי תנאי לקיפאון
**השאלה:** כביש צר מאפשר מעבר של רכב אחד בלבד. שני רכבים נכנסים משני הכיוונים ונפגשים באמצע. הם תקועים. איזה תנאי של קיפאון בא לידי ביטוי בכך שהכביש מאפשר מעבר של רכב אחד בלבד?
**ההיגיון:** נזכר בסיפור הפועלים - אי אפשר לחלוק את הפטיש.
**תשובה:** תנאי **המניעה ההדדית (Mutual Exclusion)**. המשאב (הכביש) אינו ניתן לשיתוף באותו רגע.

---

### תרגיל 2: שבירת קיפאון
**השאלה:** החלטנו לחוקק חוק: אם לפועל חסר כלי שהוא צריך, הוא חייב להניח על השולחן את הכלים שכבר יש לו ביד, ורק אז לבקש את הכלי החדש. איזה מתנאי הקיפאון שברנו?
**ההיגיון:** הפועל כבר לא יכול "להחזיק את הפטיש ובינתיים לחכות למסמר". 
**תשובה:** שברנו את תנאי **ההחזקה והמתנה (Hold and Wait)**.

---

### תרגיל 3: מציאת ה-Need (חישוב הצורך)
**השאלה:** קבלן 1 הצהיר ביום הראשון שיצטרך מקסימום 10 מיליון ש"ח (`Max=10`). עד היום הוא קיבל הלוואות בסך 4 מיליון ש"ח (`Allocation=4`). כמה הוא עוד צריך?
**ההיגיון:** נוסחת הזהב.
**תשובה:** $Need = 10 - 4 = \mathbf{6}$. (הצורך שלו הוא 6 מיליון).

---

### תרגיל 4: חישוב הקופה (Available)
**השאלה:** לבנק יש בסך הכל 15 מיליון ש"ח בכל המערכת. קבלן א' קיבל 3, קבלן ב' קיבל 2, קבלן ג' קיבל 5. כמה נשאר זמין בקופה?
**ההיגיון:** נחסר מסך הכל (Total) את סך כל ההקצאות.
**פתרון:** $15 - (3 + 2 + 5) = 15 - 10 = 5$.
**תשובה:** בקופה יש $Available = 5$.

---

### תרגיל 5: סימולציה ראשונה - האם אפשר לסיים?
**השאלה:** בקופה יש 5 מיליון. הצורך (`Need`) של קבלן א' הוא 7. הצורך של קבלן ב' הוא 4. את מי הבנקאי יכול לקדם עכשיו?
**ההיגיון:** הבנקאי משווה את ה-Need לקופה.
**פתרון:** לא' חסר 7, ולבנקאי יש רק 5 - אי אפשר לעזור לו! ל-ב' חסר 4, ולבנקאי יש 5 - אפשר בהחלט לתת לו את המימון כדי שיסיים את העבודה.
**תשובה:** הבנקאי יאשר ויקדם את קבלן ב'.

---

### תרגיל 6: עדכון הקופה לאחר סיום
**השאלה:** המשך לתרגיל 5. קבלן ב' קיבל את ה-4 שחסרו לו וסיים את הבנייה. ידוע שמלכתחילה קבלן ב' החזיק כבר 3 מיליון (`Allocation=3`). מה יהיה המצב החדש של הקופה (`Available`) ברגע ש-ב' עוזב את הבנק?
**ההיגיון:** כשקבלן מסיים, הוא מוכר את הבניין ומחזיר לבנק את **כל** מה שהוא אי פעם לקח (את ה-Allocation שלו). הבנק גם חסך את ה-Need שנתן לו הרגע (כי זה היה רק בדמיון שלנו, אבל תכלס זה חוזר לבנק). הכלל: קופה חדשה = קופה ישנה (לפני שנתנו לו) + ה-Allocation הישן שלו.
**פתרון:** הקופה הייתה 5. קבלן ב' מסיים ומחזיר את ה-3 שהוא החזיק. $5 + 3 = 8$.
**תשובה:** הקופה המעודכנת תהיה 8 מיליון.

---

### תרגיל 7: מציאת Safe Sequence (סדר בטוח)
**השאלה:** נתון המצב הבא. `Available = 3`.
*   תהליך P1: `Allocation=2`, `Need=4`.
*   תהליך P2: `Allocation=1`, `Need=2`.
*   תהליך P3: `Allocation=3`, `Need=6`.
האם המצב בטוח? (הוכח על ידי סדר ריצה).
**ההיגיון:** הבנקאי מחפש את מי אפשר להציל עם ה-3 שיש בקופה, ואז אוסף את הכסף שלו כדי להציל את הבא.
**פתרון:**
1. קופה = 3. 
   P1 צריך 4 (לא מספיק). 
   P2 צריך 2 (**מספיק!**).
2. P2 רץ. הוא מסיים ומחזיר את ה-Allocation שלו (1) לקופה.
   קופה חדשה: $3 + 1 = 4$.
3. קופה = 4. 
   P1 צריך 4 (**מספיק!**). 
   P1 רץ. מסיים ומחזיר את ה-Allocation שלו (2).
   קופה חדשה: $4 + 2 = 6$.
4. קופה = 6. 
   P3 צריך 6 (**מספיק!**). רץ ומסיים.
**תשובה:** המצב בטוח. סדר ההרצה הוא `<P2, P1, P3>`.

---

### תרגיל 8: מצב לא בטוח (Unsafe)
**השאלה:** מה היה קורה בתרגיל 7 אם ה-Need של P1 היה 5 במקום 4?
**הפתרון:**
1. קופה = 3. נותנים ל-P2. P2 מסיים ומחזיר 1.
2. קופה חדשה = 4.
3. כעת, P1 צריך 5 (יש לנו רק 4, אי אפשר). P3 צריך 6 (אי אפשר).
4. נתקענו! הבנקאי לא יכול לעזור לאף אחד מהם לסיים. המערכת תחכה לנצח.
**תשובה:** זהו מצב **Unsafe**. (למנהל בנק חכם אסור היה בכלל להגיע למצב ההתחלתי הזה!).

---

### תרגיל 9: מטריצות - סוגי משאבים מרובים (רמת מבחן פתוחה) 👑
**השאלה:** במחשב יש מדפסות, כוננים וזיכרון (נקרא להם A, B, C). לכן הנתונים הם רצפים (וקטורים).
קופה (Available): `[3, 3, 2]`
תהליך P1: מחזיק (Allocation): `[2, 0, 0]`. צריך (Need): `[1, 2, 2]`.
תהליך P2: מחזיק (Allocation): `[3, 0, 2]`. צריך (Need): `[4, 1, 1]`.
מי יכול לרוץ ראשון בסימולציה של הבנקאי?
**ההיגיון:** הבנקאי משווה את ה"סל" שהלקוח צריך ל"סל" שיש בקופה. חייב להיות מספיק מ**כל סוג משאב בנפרד**.
**פתרון:**
*   P1 צריך 1 מ-A, 2 מ-B, 2 מ-C. בקופה יש 3 מ-A, 3 מ-B, 2 מ-C.
    האם `[1,2,2]` קטן או שווה ל-`[3,3,2]` בכל העמודות? **כן!** P1 יכול לרוץ.
*   P2 צריך 4 מ-A. בקופה יש רק 3 מ-A. 
    האם `[4,1,1]` קטן או שווה ל-`[3,3,2]`? **לא.** (4 גדול מ-3). P2 לא יכול לרוץ.
**תשובה:** תהליך P1 חייב לרוץ ראשון.

---

### תרגיל 10: בקשה פתאומית (Request) ממבחן
**השאלה:** המשך לתרגיל 9. תהליך P1 אומר: "במקום לחכות, תן לי הרגע באופן מיידי בקשה של `[1, 0, 2]` משאבים". מה הבנקאי עושה?
**ההיגיון:** הבנקאי מבצע 3 בדיקות לפני שהוא נותן את הכסף: "האם זה חוקי? האם יש לי בקופה? האם זה משאיר אותי בטוח?".
**הפתרון השלבי:**
1.  **בדיקת שקרן:** האם הבקשה `[1,0,2]` קטנה או שווה ל-Need המקורי שלו `[1,2,2]`? כן. (הוא לא מבקש יותר ממה שהצהיר).
2.  **בדיקת קופה:** האם הבקשה `[1,0,2]` קטנה או שווה למה שיש ב-Available `[3,3,2]`? כן. (יש מספיק לתת לו).
3.  **סימולציית העתיד:** הבנקאי אומר - נניח ונתתי לו. 
    *   הקופה תרד: `[3,3,2] - [1,0,2] = [2, 3, 0]`.
    *   ההקצאה של P1 תגדל: `[2,0,0] + [1,0,2] = [3, 0, 2]`.
    *   הצורך של P1 יקטן (כי הוא קיבל): `[1,2,2] - [1,0,2] = [0, 2, 0]`.
4.  עכשיו מריצים את אלגוריתם הבטיחות מההתחלה על המספרים החדשים כדי לוודא שלא נתקענו. (אם נגלה שאפשר לסיים - הבקשה תאושר!).

# סיכום קורס מערכות הפעלה (20594) - וירטואליזציה וקונטיינרים (שאלות 11-12)

הנושא הזה עוסק בדרך שבה אנחנו נותנים למספר תוכנות או לקוחות לרוץ על מחשב פיזי אחד, מבלי שהם ידעו אחד על השני, וכדי שלא יהרסו אחד לשני את הקבצים.

## 1. המושגים והסיפור האחיד: "הבניין והשכירות" 🏢🏠

דמיין שהמחשב הפיזי שלך הוא **חלקת אדמה ענקית עם תשתיות חשמל ומים (החומרה)**. אנחנו רוצים להושיב עליה הרבה לקוחות. יש שתי דרכים לעשות את זה:

### א. מכונה וירטואלית (VM - Virtual Machine) = "לבנות בית פרטי חדש"
*   **ההסבר:** לכל לקוח אנחנו בונים בית פרטי שלם מאפס. יש לו יסודות משלו, קירות משלו ומערכת חשמל משלו (מערכת הפעלה מלאה משלו - Guest OS). 
*   **יתרון:** בידוד מוחלט! הלקוח יכול להחליט שהוא רוצה בית מעץ (Windows) או בית מלבנים (Linux). 
*   **חיסרון:** זה כבד, יקר, ולוקח המון זמן לבנות ולתחזק. זה מבזבז המון זיכרון (RAM) רק כדי להחזיק את הבית עומד.

### ב. קונטיינר (Container) = "לשכור דירה בבניין קיים"
*   **ההסבר:** במקום לבנות בתים פרטיים, אנחנו בונים בניין דירות אחד ענק (מערכת הפעלה אחת - Host OS). לכל לקוח נותנים מפתח לדירה אטומה משלו. כולם שותים מאותו צינור מים מרכזי (ה-Kernel - ליבת המערכת), אבל אף אחד לא יכול לראות מה השכן עושה.
*   **יתרון:** סופר-מהיר וקל משקל! הלקוח נכנס לדירה והיא מוכנה בשנייה. לא מבזבז כמעט זיכרון.
*   **חיסרון:** כולם חייבים להשתמש באותה תשתית. אם הבניין הוא בניין של Linux, אי אפשר פתאום להריץ בדירה תוכנה שדורשת Windows.

### ג. היפרוויזר (Hypervisor) = "קבלן הבנייה של ה-VMs"
*   **Type 1 (קבלן ישיר - Bare Metal):** הקבלן בונה את הבתים ישירות על חלקת האדמה. (הכי מהיר ויעיל, ככה עובדים בשרתים של אמזון וגוגל).
*   **Type 2 (קבלן על גגות - Hosted):** על האדמה כבר יש בניין רגיל (למשל ה-Windows 10 בבית שלך), והקבלן מוסיף עליו "קומות וירטואליות". (פחות יעיל, מיועד למחשבים אישיים).

### ד. פקודת ה-`unshare` = "בניית דירה אטומה חדשה"
כשתהליך עושה `unshare`, הוא אומר למערכת: "אני רוצה להקים קונטיינר חדש". 

---

## 2. חוקי הברזל למבחן (לקוד של שאלה 11)

1.  **חוק הלידה (`fork`):** תהליך הורה מפעיל `pid = fork()`. נולד ילד. 
    *   האבא מסתכל על המשתנה `pid` ורואה בו את תעודת הזהות (ה-PID) של הילד החדש (למשל 50).
    *   הילד מסתכל על המשתנה `pid` ורואה **0**. ככה הוא מבין שהוא הילד!
2.  **חוק ה"מחוץ לבועה":** פקודת `unshare` לא מכניסה את מי שכתב אותה לתוך הבועה! היא רק מכינה את הבועה ל**ילדים שייוולדו לו מעכשיו והלאה**. האבא נשאר בחוץ.
3.  **חוק "האדם הראשון":** הילד הראשון שנולד לתוך בועת קונטיינר חדשה יהיה תמיד **PID מספר 1** בתוך הבועה הזו.
4.  **חוק היתמות (`getppid`):** אם הילד שבתוך הבועה מנסה לשאול "מי האבא שלי?" (`getppid()`), אבל האבא נשאר מחוץ לבועה - מערכת ההפעלה של הקונטיינר לא מכירה את האבא! הילד יקבל את התשובה **0**.

---

## 3. תרגול: מאפס לרמת מבחן

### תרגיל 1: שאלת תיאוריה קלאסית (VM מול Container)
**השאלה:** הסבר מדוע הפעלת קונטיינר מהירה בהרבה מהפעלת מכונה וירטואלית (VM)?
**ההיגיון:** מה מהיר יותר? להדליק אור בדירה שכורה או לבנות בית מאפס ולחבר אותו לחשמל?
**הפתרון:** במכונה וירטואלית (VM), בכל הפעלה צריך להעלות מחדש (Boot) מערכת הפעלה שלמה מאפס, מה שלוקח דקות ומשאבים רבים. קונטיינר לעומת זאת משתמש בליבת מערכת ההפעלה (Kernel) שכבר רצה ברקע, ולכן הוא עולה כמעט באופן מיידי, כמו תוכנה רגילה.

### תרגיל 2: מגבלת הקונטיינר (שאלת הבנה)
**השאלה:** האם ניתן להריץ מערכת הפעלה Windows בתוך קונטיינר שרץ על שרת שמותקן בו Linux?
**ההיגיון:** האם אפשר לגור בדירה של בניין לינוקס ולהשתמש בצנרת של ווינדוס?
**הפתרון:** **לא.** קונטיינר מבודד רק סביבת עבודה, אבל חולק את ליבת המערכת (Kernel) עם המארח. קוד של Windows דורש Kernel של Windows כדי לפעול, ולכן לא יעבוד על Kernel של Linux. במכונה וירטואלית (VM) זה כן אפשרי.

### תרגיל 3: קבלנים (Hypervisors)
**השאלה:** מה ההבדל בין Hypervisor מסוג 1 ל-Hypervisor מסוג 2?
**הפתרון:** סוג 1 (Bare Metal) רץ ישירות על החומרה הפיזית ללא תיווך, מה שמעניק לו ביצועים טובים מאוד. סוג 2 (Hosted) רץ כתוכנה רגילה מעל מערכת הפעלה קיימת (למשל Windows המותקן במחשב), ולכן יש לו תקורת (Overhead) ביצועים של מערכת ההפעלה המארחת.

### תרגיל 4: המספרים של `fork`
**השאלה:** תהליך שמספרו 10 מריץ את הפקודה `pid = fork()`. נניח שהילד קיבל את המספר 11. מה יהיה מודפס בשורה `printf("%d", pid)` אם האבא מריץ אותה, ומה אם הילד מריץ אותה?
**ההיגיון:** האבא מקבל את הת"ז של הילד. הילד מקבל 0.
**הפתרון:** האבא ידפיס 11. הילד ידפיס 0.

### תרגיל 5: הקסם של `unshare`
**השאלה:** תהליך מריץ את הפקודה `unshare(PID_NS)`. האם מספר ה-PID של התהליך הזה משתנה ל-1 בעקבות הפקודה?
**ההיגיון:** האבא שבנה את הדירה לא נכנס אליה! הוא נשאר בחוץ.
**הפתרון:** **לא.** פקודת `unshare` משפיעה רק על **ילדים** שייווצרו לאחר מכן בעזרת `fork`. התהליך המקורי נשאר באותו מרחב שמות ובאותו PID שהיה לו קודם.

### תרגיל 6: הלידה אל תוך הבועה
**השאלה:** תהליך אבא (PID 20) מבצע את הקוד הבא:
`unshare(PID_NS);`
`fork();`
מהו ה-PID של הילד החדש בעולם הגלובלי מחוץ לבועה, ומהו ה-PID של הילד מנקודת המבט שלו עצמו בתוך הבועה?
**הפתרון:** בעולם הגלובלי הילד הוא תהליך רגיל שיקבל את המספר הבא (PID 21). בתוך הבועה (הקונטיינר) שהוא נולד אליה, הוא התהליך הראשון אי פעם! לכן ה-PID שלו יהיה **1**.

### תרגיל 7: השאלה "מי אבא שלי?" (`getppid`)
**השאלה:** הילד מתרגיל 6 מבצע את הפקודה `getppid()` (בקשה לקבל את מספר ה-PID של האבא שלו). מה הוא יקבל ולמה?
**ההיגיון:** הילד בבועה, האבא בחוץ. הילד מסתכל סביב ולא רואה את אבא.
**הפתרון:** הפונקציה תחזיר **0**. מכיוון שהאבא נמצא מחוץ ל-Namespace המבודד, הוא "בלתי נראה" עבור הילד. לכן הילד לא רואה אבא חוקי במרחב שלו.

### תרגיל 8: קוד מבחן מלא (חלק 1)
**השאלה:** ה-PID הגבוה ביותר במערכת לפני תחילת הקוד היה 5. מה ידפיס הקטע הבא?
```c
unshare(PID_NS);
pid = fork();
if (pid > 0) {
    printf("Parent sees child as: %d", pid);
}
```
**הפתרון צעד-אחר-צעד:**
1. הפעילו `unshare`. הבועה מוכנה לדור הבא.
2. עשו `fork`. המקסימום היה 5, לכן נולד ילד עם PID גלובלי של **6**.
3. האבא נכנס ל-`if` (כי בשבילו `pid = 6` שזה גדול מ-0). 
4. האבא מדפיס את ה-PID הגלובלי שהוא מכיר: `Parent sees child as: 6`.

### תרגיל 9: קוד מבחן מלא (חלק 2 - מתוך מבחן 68) 👑
**השאלה:** ה-PID הגבוה לפני תחילת הקוד היה 10.
```c
unshare(PID_NS);
pid = fork();
if (pid == 0) { // זהו הילד!
    printf("Child sees himself as: %d", getpid());
    printf("Child sees parent as: %d", getppid());
}
```
**הפתרון צעד-אחר-צעד:**
1. הילד (PID גלובלי 11) נכנס ל-`if`.
2. הוא שואל מה ה-PID שלו עצמו (`getpid`). כיוון שהוא הראשון בתוך הבועה שנוצרה ב-`unshare`, הוא יענה **1**.
3. הוא שואל מה ה-PID של אבא שלו (`getppid`). בגלל שאבא נשאר מחוץ לבועה, האבא בלתי נראה, ולכן התשובה היא **0**.
**תשובה:**
`Child sees himself as: 1`
`Child sees parent as: 0`

### תרגיל 10: השפעת המיקוד החדש במבחנים (MOUNT vs PID)
**השאלה:** במבחן אחר נכתב בקוד: `unshare(MOUNT_NS)` במקום `PID_NS`, ולאחר מכן יש פתיחה של קבצים (יצירת `dirS`). איך מתייחסים לזה?
**ההיגיון:** קוראים את המיקוד שהמרצה פרסמה!
**הפתרון:** המיקוד אומר מפורשות: "במבחן לא תהיה התייחסות להבדלים בין מרחבי שמות (Namespaces) שונים". המשמעות היא שעבורך במבחן, `unshare` פשוט אומר: "הילד נכנס לעולם מבודד משלו". 
אם הילד יצר תיקייה, האבא בחוץ לא יראה אותה! אם האבא ינסה לפתוח את התיקייה הזו, הוא יקבל שגיאה (Error) כי בעולם של האבא היא לא קיימת. (וזה בדיוק הפתרון לשאלה במבחן 519 שסרקנו).

# סיכום קורס מערכות הפעלה (20594) - תהליכים, תזמון ומושגי יסוד (חלק א')

הנושא הזה עוסק בשאלה: כשיש לנו רק מעבד אחד (או מעט מעבדים) אבל עשרות תוכניות שרוצות לרוץ במקביל, איך המחשב מחליט מי ירוץ, מתי, ולכמה זמן?

## 1. המושגים והסיפור האחיד: "מטבח המסעדה" 🍳

דמיין שמערכת ההפעלה היא **מסעדה עמוסה**.
*   **המעבד (CPU) = "השף":** במסעדה שלנו יש רק שף אחד! הוא יכול להכין רק מנה אחת בדיוק בכל רגע נתון.
*   **תהליך (Process) = "הזמנה משולחן":** יישות עצמאית וגדולה. למשל, "שולחן 4 הזמין ארוחה". לכל הזמנה יש "חשבון" משלה (זיכרון פרטי) שאסור לערבב עם שולחנות אחרים.
*   **תהליכון (Thread) = "מנה בתוך ההזמנה":** בתוך ההזמנה של שולחן 4 יש צ'יפס, סלט והמבורגר. אלו תהליכונים. כולם שייכים לאותה הזמנה, כולם נרשמים על אותו חשבון (חולקים זיכרון), אבל אפשר לעבוד עליהם בנפרד.
*   **החלפת הקשר (Context Switch) = "שטיפת ידיים והחלפת סינר":** כשהשף עוצר באמצע הכנת בשר כדי לחתוך סלט לשולחן אחר, הוא חייב לנקות את הקרש, לשטוף ידיים, לזכור איפה הוא עצר בבשר, ולהוציא מתכון לסלט. **זו פעולה שלוקחת זמן (תקורה - Overhead) ולא מקדמת את האוכל!**
*   **מתזמן (Scheduler) = "מנהל המשמרת":** הוא עומד עם ה"בונים" (הפתקים של ההזמנות) ומחליט איזה פתק לתת לשף עכשיו.
*   **קלט/פלט (I/O) = "שליחויות / חומרי גלם":** פעולות איטיות שקורות מחוץ למטבח (כמו הבאת מלח מהמחסן, או הדפסה למסך).
*   **TRAP (פסיקת תוכנה) = "קריאה למפקח משרד הבריאות":** כשמנה צריכה משהו מסוכן או שדורש סמכות עליונה, עוצרים הכל, עוברים ל"מצב מנהל" (Kernel Mode) ומטפלים בזה.

---

## 2. מצבי תהליך (Process States)

מנה (תהליך) לא יכולה להיות כל הזמן אצל השף. היא עוברת בין 3 מצבים עיקריים:
1.  **רץ (Running):** השף בפועל חותך, מטגן ומכין את המנה עכשיו.
2.  **מוכן (Ready):** המנה מוכנה להכנה, הפתק תלוי על הלוח, אבל השף כרגע מכין מנה אחרת. המנה מחכה בתור.
3.  **חסום / ממתין (Blocked / Waiting):** חסר מלח! השף לא יכול להמשיך. הוא מוריד את המנה מהאש ושם אותה בצד עד שהשליח יחזור עם המלח (ה-I/O יסתיים). ברגע שהמלח יגיע, המנה לא תחזור ישר לשף, אלא תעבור למצב **מוכן (Ready)** ותחכה לתורה שוב.

---

## 3. אלגוריתמי התזמון של מנהל המשמרת

איך מנהל המשמרת בוחר את הפתק הבא?
*   **FCFS (First Come First Served):** מי שהזמין קודם, מקבל קודם. פשוט, אבל אם שולחן 1 הזמין מנה שלוקחת 10 שעות, כל שאר המסעדה תתייבש בתור (אפקט השיירה).
*   **SJF (Shortest Job First):** המנה שתיקח הכי מעט זמן תוכַן ראשונה. **זה האלגוריתם שנותן את זמן ההמתנה הממוצע הנמוך ביותר!**
*   **SRTF (Shortest Remaining Time First):** כמו SJF, אבל "אגרסיבי" (Preemptive). אם השף מכין מנה של שעה, ופתאום נכנסת מנה של דקה - השף יזרוק הכל, יכין את המנה של הדקה, ואז יחזור לקודמת!
*   **Round Robin (RR):** תזמון דמוקרטי. מנהל המשמרת נותן לשף **קצבת זמן (Quantum)** של דקה למנה. דקה להמבורגר, עוזב. דקה לסלט, עוזב. כולם מרגישים שמתייחסים אליהם במקביל.

## 4. 10 תרגילים מדורגים (מאפס לרמת מבחן אמריקאי)

### תרגיל 1: הבדל בין תהליך (Process) לתהליכון (Thread)
**השאלה:** אילו מהמשאבים הבאים משותפים לכל התהליכונים (Threads) השייכים לאותו תהליך?
א. משתנים מקומיים (מחסנית/Stack).
ב. מרחב הזיכרון (קוד ונתונים גלובליים).
**ההיגיון:** האם טבחים שעובדים על אותה הזמנה חולקים את אותו חשבון לקוח? כן. האם הם חולקים את אותה סכין חיתוך בדיוק באותה שנייה? לא.
**הפתרון:** כל התהליכונים חולקים את אותו זיכרון וקבצים פתוחים. אבל לכל תהליכון יש מחסנית (Stack) משלו ו-Program Counter משלו (מצביע לשורת הקוד שהוא מבצע כרגע).
**תשובה: ב'. מרחב הזיכרון משותף לכולם.**

---

### תרגיל 2: מעברי מצבים (State Transitions)
**השאלה:** תהליך רץ על המעבד (Running) ופתאום מבצע בקשה לקריאת קובץ מהדיסק הקשיח (פעולת קלט/פלט ארוכה). לאיזה מצב יעבור התהליך?
**ההיגיון:** השף מגלה שחסר מלח והזמין משלוח. מה הוא עושה עם המנה? שם אותה בתור ה"מוכנים"? לא, היא אי אפשר להמשיך להכין אותה.
**הפתרון:** המערכת משהה את התהליך ומעבירה אותו למצב שבו הוא לא מתחרה על המעבד עד שהמידע יגיע.
**תשובה: מצב חסום (Blocked / Waiting).**

---

### תרגיל 3: תהליך זומבי (Zombie Process) - קלאסיקה!
**השאלה (מתוך מבחן אמיתי):** בגרף המצבים של תהליך בלינוקס, תהליך שמסיים את עבודתו עובר למצב "זומבי" (Zombie). מהי הסיבה לכך?
**ההיגיון:** ילד סיים לאכול במסעדה. הוא רוצה ללכת, אבל אסור לו לצאת מהדלת עד שהאבא שלו לא יבוא לקרוא את קבלת התשלום שלו. 
**הפתרון:** תהליך שמסיים לעבוד (עשה `exit`) הופך לזומבי. המערכת שומרת עליו כ"מת-חי" רק כדי לשמור את נתוני הסיום שלו (Exit Status). הוא ייעלם סופית רק כשתהליך ה"אבא" שלו יבצע את הפקודה `wait` ויקרא את הנתונים האלו.
**תשובה: כדי לשמור מידע על התהליך שסיים עד שהאבא שלו יבצע `wait`.**

---

### תרגיל 4: Round Robin וגודל ה-Quantum
**השאלה:** מה יקרה באלגוריתם Round Robin אם נגדיר את קצבת הזמן (ה-Quantum) להיות קטנה מאוד (למשל, 1 מילי-שנייה)?
**ההיגיון:** מה יקרה אם השף יחליף מנה כל שנייה אחת בדיוק? הוא ישטוף ידיים כל שנייה.
**הפתרון:** כל החלפה דורשת החלפת הקשר (Context Switch). אם הקוואנטום קטן מדי, המעבד יבלה את רוב זמנו בלהחליף תהליכים (לשמור רגיסטרים, לטעון רגיסטרים) במקום לעשות עבודה אמיתית.
**תשובה:** ביצועי המערכת יירדו משמעותית בגלל תקורה (Overhead) גבוהה של החלפות הקשר.

---

### תרגיל 5: חישוב זמן המתנה ב-SJF (חישוב AWT)
**השאלה:** שלושה תהליכים (P1, P2, P3) מגיעים למערכת יחד בזמן 0.
זמן הריצה של P1 הוא 10 שניות.
זמן הריצה של P2 הוא 2 שניות.
זמן הריצה של P3 הוא 5 שניות.
מה יהיה ממוצע זמן ההמתנה (Average Waiting Time) באלגוריתם SJF?
**ההיגיון:** SJF אומר "הקצר ביותר רץ ראשון".
**הפתרון:** 
1. סדר הרצה: P2 (הכי קצר), אחר כך P3, ואז P1.
2. נחשב כמה כל אחד *חיכה* לפני שהתחיל:
   * P2 חיכה: **0** שניות (הוא הראשון).
   * P3 חיכה ש-P2 יסיים: **2** שניות.
   * P1 חיכה ש-P2 ו-P3 יסיימו יחד: $2 + 5 =$ **7** שניות.
3. ממוצע: $(0 + 2 + 7) / 3 = 9 / 3 = \mathbf{3}$.
**תשובה: זמן ההמתנה הממוצע הוא 3 שניות.**

---

### תרגיל 6: הפקעה (Preemption) ב-SRTF
**השאלה:** אלגוריתם ה-SRTF הוא הגרסה המופקעת (Preemptive) של SJF. מה המשמעות של הפקעה בתזמון מעבד?
**ההיגיון:** "הפקעה" זה כמו שמנהל המסעדה נכנס למטבח ולוקח לשף את המחבת מהיד באמצע מנה.
**הפתרון:** הפקעה (Preemption) אומרת שלמערכת ההפעלה יש סמכות לעצור תהליך שרץ כרגע על המעבד **באמצע העבודה שלו**, להזיז אותו חזרה לתור ה"מוכנים", ולהכניס תהליך אחר במקומו (למשל כי הגיע תהליך קצר ודחוף יותר). אלגוריתמים ללא הפקעה (Non-Preemptive) נותנים לתהליך לרוץ עד שהוא מחליט לסיים לבד.
**תשובה: היכולת לקטוע תהליך רץ ולהחליפו באחר.**

---

### תרגיל 7: הפקודה `fork()` (מתמטיקה קטנה)
**השאלה:** תוכנית רצה קוד ובו 4 פקודות `fork()` רצופות (אחת אחרי השנייה). כמה תהליכים יהיו קיימים סך הכל במערכת בסוף הריצה (כולל התהליך המקורי)?
**ההיגיון:** פקודת `fork` משכפלת את התהליך הקיים לשניים. כלומר, כמות התהליכים מכפילה את עצמה בכל הפעלה.
**הפתרון:** הנוסחה היא $2^N$, כאשר $N$ הוא מספר פקודות ה-`fork`. במקרה שלנו $N=4$. לכן $2^4 = 16$.
**תשובה: יהיו 16 תהליכים.**

---

### תרגיל 8: TRAP מול פונקציה רגילה
**השאלה:** מה ההבדל המרכזי בין קריאה לפונקציית ספריה (כמו חישוב מתמטי) לבין ביצוע קריאת מערכת (System Call) שמעוררת TRAP?
**ההיגיון:** קריאה רגילה היא "לטגן חביתה בעצמך". TRAP זה לקרוא ל"חבלן משטרתי" כדי לפתוח כספת.
**הפתרון:** פונקציה רגילה מתבצעת ב-User Mode (מצב משתמש, שבו אין הרשאות מיוחדות). פקודת TRAP מעבירה את המעבד פיזית ל-Kernel Mode (מצב ליבה), מה שמעניק למערכת ההפעלה שליטה מלאה על החומרה.

---

### תרגיל 9: וירטואליזציה (Simulate and Trap) - שאלת מבחן אכזרית
**השאלה (מתוך מועד 72):** איזו תכונה חשובה, הכרחית ומספיקה למעבד (CPU) בשביל שיהיה ניתן לממש עליו וירטואליזציה יעילה בשיטת "Simulate-and-Trap"?
א. המעבד יכול לפעול גם ב-User Mode וגם ב-Kernel Mode.
ב. כל הפקודות הרגישות (Sensitive) מהוות תת-קבוצה של הפקודות המיוחסות (Privileged / גורמות ל-TRAP).
ג. יכולת לבצע פקודות רגישות קיימת רק ב-Kernel Mode.
**ההיגיון:** כדי שה"קבלן" (ההיפרוויזר) יוכל לשלוט בכל המערכות המדומות, הוא חייב שכל פעם שהן מנסות לעשות משהו מסוכן ("פקודה רגישה"), המעבד יזרוק אותן אוטומטית אליו לטיפול ("יעשה להן TRAP"). 
**הפתרון:** המשפט הפורמלי אומר: חובה שקבוצת הפקודות שמשנות את מצב המערכת (רגישות) תהיה חלק מקבוצת הפקודות שמקפיצות שגיאת הרשאה כשמריצים אותן במצב משתמש (מיוחסות).
**תשובה: ב'. כל הפקודות הרגישות חייבות להיות פקודות מיוחסות.**

---

### תרגיל 10: צוואר הבקבוק (I/O Bound vs CPU Bound)
**השאלה:** מנהל המשמרת צריך לבחור אילו תהליכים ירוצו במערכת. יש תהליכים שעושים המון חישובים (CPU Bound) ויש תהליכים שמבלים את רוב הזמן בקריאה מהדיסק (I/O Bound). איזה תמהיל (שילוב) יעדיף מנהל המשמרת להכניס לזיכרון?
**ההיגיון:** השף רוצה לבשל, אבל הוא לא יכול אם כל המנות הן כאלו שצריכות מצרכים מהמחסן (הכל נתקע בשליח). מנגד, גם השליח צריך לעבוד.
**הפתרון:** כדי לנצל את המערכת בצורה המקסימלית, המתזמן שואף לתמהיל **מעורב** (Mix). כך, בזמן שהתהליכים מוטי ה-I/O ממתינים לדיסק הקשיח (נמצאים במצב Blocked), השף (ה-CPU) פנוי להריץ במלוא המרץ את התהליכים מוטי ה-CPU שמוכנים לעבוד.
**תשובה: שילוב מעורב של תהליכי I/O ותהליכי CPU.**
