

# סיכום קורס מערכות הפעלה (20594) - סנכרון תהליכים (שאלה 7)

כאן ריכזתי את כל מה שלמדתי על הנושא הכי "רווחי" במבחן – **שאלה 7 (סנכרון)**. שאלה זו שווה בדרך כלל בין 23 ל-26 נקודות. המטרה היא להבין את הלוגיקה מאפס, דרך סיפורים והדמיות, וליישם תבנית פתרון קבועה.

## 1. מושגי יסוד - השפה של הסנכרון

לפני שניגשים לקוד, חייבים להבין את אבני הבניין:

*   **משאב משותף (Shared Resource):** כל דבר (משתנה, תור, מחסן, קובץ) שיותר מתהליך אחד רוצה לגשת אליו בו-זמנית.
*   **מרוץ קריטי (Race Condition):** המצב הרע שקורה כששני תהליכים משנים משאב משותף יחד, והתוצאה הסופית משתבשת (כמו שני אנשים שמושכים כסף מאותו חשבון בנק באותה שניה).
*   **קטע קריטי (Critical Section):** שורות הקוד שבהן אנחנו נוגעים במשאב המשותף. המטרה שלנו: להבטיח שרק תהליך אחד נמצא שם בכל רגע.
*   **סמפור (Semaphore):** "קופסת מפתחות". משתנה מיוחד ששולט בגישה:
    *   `Down` (או `wait` / `P`): פעולה של "לקחת מפתח". אם הקופסה ריקה, התהליך נרדם ומחכה.
    *   `Up` (או `signal` / `V`): פעולה של "להחזיר מפתח". אם מישהו מחכה בחוץ, הוא מתעורר מיד.
*   **מיוטקס (Mutex):** סמפור עם מפתח **אחד בלבד**. משמש כ"מנעול" (0 = תפוס, 1 = פנוי).
*   **קיפאון (Deadlock):** מצב שבו תהליך א' מחכה לתהליך ב', ותהליך ב' מחכה לתהליך א' – ושניהם תקועים לנצח.

## 2. נושא א': בעיית יצרן-צרכן (Producer-Consumer)

### שאלה 1 (מבוסס על מבחן 2024 מועד 68 / 2026 מועד 91)
**תיאור השאלה:**
נתון מחסן (Buffer) בגודל $N$ המנוהל כרשימה מקושרת. הפונקציות `Enqueue` (הכנסה) ו-`Dequeue` (הוצאה) אינן אטומיות.
יש להשלים את הקוד עבור יצרן בודד ומספר צרכנים, תוך שימוש בסמפורים הבאים:
*   `BufferState1`: סמפור מניה (Counting).
*   `BufferState2`: סמפור מניה (Counting).
*   `mutex1`: סמפור בינארי (מנעול).
*   `mutex2`: סמפור בינארי (מנעול).

### סיפור ההדמיה: "המסוע של הסושי" 🍣
דמיין מסוע סושי עם **N** מקומות פנויים.
*   **השף (יצרן):** מכין סושי. הוא לא יכול להניח סושי אם המסוע מלא.
*   **הלקוח (צרכן):** אוכל סושי. הוא לא יכול לאכול אם המסוע ריק.
*   **הסמפורים הם קערות כרטיסים:** 
    *   `empty`: כמה מקומות פנויים נשארו? (מתחיל ב-N).
    *   `full`: כמה מנות מוכנות יש? (מתחיל ב-0).
    *   `mutex`: שלט שאומר "אני נוגע במסוע". רק מי שמחזיק אותו יכול להוסיף/להוציא מנה.

### תבנית הפתרון (הצעדים האבסולוטיים)
1.  **קודם כל בודקים משאב:** (יש מקום? יש אוכל?) ע"י `down` לסמפור המניה.
2.  **נועלים את המנעול:** `down(mutex)`.
3.  **מבצעים את הפעולה:** `Enqueue` / `Dequeue`.
4.  **משחררים מנעול:** `up(mutex)`.
5.  **מעדכנים את הצד השני:** `up` לסמפור המניה השני.

### תשובה מלאה:
```c
// אתחול:
BufferState1 = N; // מייצג מקומות פנויים (empty)
BufferState2 = 0; // מייצג מוצרים מוכנים (full)

ProducerFunction(item) {
    down(BufferState1); // חכה למקום פנוי
    down(mutex1);       // נעל את המחסן
    Enqueue(item);      // הוסף מוצר
    up(mutex1);         // שחרר נעילה
    up(BufferState2);   // תגיד לצרכן שיש מוצר
}

ConsumerFunction() {
    down(BufferState2); // חכה למוצר מוכן
    down(mutex1);       // נעל את המחסן
    item = Dequeue();   // קח מוצר
    up(mutex1);         // שחרר נעילה
    up(BufferState1);   // תגיד ליצרן שהתפנה מקום
    return item;
}
```

## 3. נושא ב': בעיית קוראים-כותבים (Readers-Writers)

### שאלה 2 (מבוסס על מבחן 2024 מועד 72)
**תיאור השאלה:**
במערכת יש קוראים וכותבים הניגשים למשאב משותף. קוראים יכולים לקרוא יחד, אך כותב חייב להיות לבד.
יש לתת **העדפה לכותבים**: אם כותב מחכה, קוראים חדשים לא יכולים להיכנס.
**הסמפורים הנתונים:**
*   `BlockRead`: חוסם קוראים.
*   `BlockWrite`: מאפשר כתיבה של כותב אחד או קריאה של קוראים רבים.
*   `RMutex`, `WMutex`: מגנים על המונים `Readers` ו-`Writers`.

### סיפור ההדמיה: "הספרייה והשומר בשער" 📚🚪
דמיין ספרייה עם ספר נדיר אחד.
*   **הקוראים:** יכולים לקרוא בבת אחת. הקורא הראשון שנכנס נועל את הדלת בפני כותבים. הקורא האחרון שיוצא פותח את הדלת.
*   **הכותבים:** צריכים שקט מוחלט. ברגע שכותב מגיע, הוא אומר ל**שומר בשער** (`BlockRead`): "אל תכניס יותר קוראים חדשים לבניין!". הוא מחכה שמי שכבר בפנים יסיים, נכנס, כותב, ורק כשהכותב האחרון מסיים הוא משחרר את השומר בשער.

### חוק ה"ראשון והאחרון"
זהו כלל אבסולוטי לבעיות עם מונים:
*   **הראשון (count == 1):** מבצע פעולת חסימה על הקבוצה השנייה.
*   **האחרון (count == 0):** מבצע פעולת שחרור לקבוצה השנייה.

### תשובה מלאה:

**הקורא (Reader):**
```c
Reader {
    down(BlockRead);    // שומר בשער: האם כותב חוסם כניסה?
    down(RMutex);       // נעל מונה קוראים
    Readers++;
    if (Readers == 1) 
        down(BlockWrite); // הראשון נועל את הספר בפני כותבים
    up(RMutex);         // שחרר מונה
    up(BlockRead);      // שחרר את השומר בשער עבור קוראים אחרים

    < Do the Reading >

    down(RMutex);       // נעל מונה כדי לעדכן
    Readers--;
    if (Readers == 0) 
        up(BlockWrite);   // האחרון פותח את הספר לכותבים
    up(RMutex);
}
```

**הכותב (Writer):**
```c
Writer {
    down(WMutex);       // נעל מונה כותבים
    Writers++;
    if (Writers == 1) 
        down(BlockRead);  // הכותב הראשון נועל את השומר בשער (עדיפות!)
    up(WMutex);

    down(BlockWrite);   // חכה שהספר יתפנה ונעל אותו
    < Do the Writing >
    up(BlockWrite);     // שחרר את הספר

    down(WMutex);
    Writers--;
    if (Writers == 0) 
        up(BlockRead);    // הכותב האחרון משחרר את השומר בשער
    up(WMutex);
}
```

## 4. טיפים ודברים שחוזרים על עצמם (Checklist למבחן)

1.  **חוק הסנדוויץ':** כל פעם שמעדכנים משתנה (যেমন `++` או `--`), זה חייב להיות בין `down(mutex)` ל-`up(mutex)`.
2.  **מניעת Deadlock:** ב"יצרן-צרכן", תמיד בצע `down` על המשאב (מקום/מוצר) **לפני** ה-`down` על ה-`mutex`.
3.  **עדיפות:** אם מבקשים עדיפות לקבוצה מסוימת, היא חייבת לנעול סמפור שמופיע בשורה הראשונה של הקבוצה השנייה.
4.  **אטומיות:** אם רשום שפונקציה היא "לא אטומית", זה רמז עבה שחייבים להשתמש במיוטקס סביבה.
5.  **בדיקת תנאי:** בבעיות כמו "פילוסופים" או "סועדים", בדוק תמיד אם התנאי הוא "הראשון נועל" או שכל אחד נועל לעצמו.

# סיכום קורס מערכות הפעלה (20594) - ניהול זיכרון ותרגום כתובות (שאלה 8)

נושא זה עוסק בדרך שבה המחשב מתרגם כתובת "וירטואלית" (מה שהתוכנה רואה) לכתובת "פיזית" (המיקום האמיתי ב-RAM). בגלל המיקוד, אנחנו מתמקדים רק בחישובים הטכניים ובמבנה הטבלאות.

## 1. מושגי יסוד - השפה של הזיכרון

*   **זיכרון וירטואלי (Virtual Memory):** "מרחב דמיוני" שנותנים לכל תהליך, כדי שיחשוב שיש לו המון זיכרון רציף.
*   **דף (Page):** יחידת חלוקה לוגית של הזיכרון הווירטואלי (למשל, חתיכות של 4KB).
*   **מסגרת (Frame):** המקבילה הפיזית של הדף בתוך ה-RAM. גודל המסגרת תמיד שווה לגודל הדף.
*   **היסט (Offset):** המיקום המדויק בתוך הדף. אם דף הוא עמוד בספר, ה-Offset הוא מספר השורה. **הוא לעולם לא משתנה בתרגום.**
*   **טבלת דפים (Page Table):** המפה (התרגום). היא אומרת למעבד: "דף מספר X של התוכנה נמצא כרגע במסגרת מספר Y ב-RAM".
*   **PTE (Page Table Entry):** שורה אחת בטבלה. מכילה את מספר המסגרת וביטים של בקרה (כמו ביט Valid).
*   **ביט נוכחות (Valid Bit):** סימון ב-PTE. אם הוא '1' - הדף ב-RAM. אם הוא '0' - הדף בדיסק (זה גורם ל-**Page Fault**).

### המתמטיקה שחובה לזכור:
הכל עובד בחזקות של 2:
*   $1KB = 2^{10} Bytes$
*   $4KB = 2^{12} Bytes$ (גודל דף סטנדרטי)
*   $1MB = 2^{20} Bytes$
*   $1GB = 2^{30} Bytes$

## 2. שאלה 1 ממבחן: חלוקת כתובת וירטואלית (מבנה טבלה רב-שלבית)

**תיאור השאלה:**
נתונה כתובת וירטואלית באורך **32 סיביות**. המערכת משתמשת בטבלת דפים ב-**2 רמות**. אורך כל כניסה בטבלה (PTE) הוא **4 בתים**.
**הדרישה:** כל טבלת דפים בודדת (הראשית והמשניות) חייבת לתפוס בדיוק **דף זיכרון אחד**. 
מהו פורמט החלוקה לסיביות של הכתובת?

### סיפור ההדמיה: "מפת העיר הענקית" 🗺️
דמיין עיר כל כך גדולה, שאי אפשר להחזיק מפה שלה בדף אחד.
1.  **האופסט (Offset):** זה מספר הבית ברחוב.
2.  **רמה 2 (טבלה משנית):** זה דף שמכיל רשימת רחובות בשכונה מסוימת.
3.  **רמה 1 (טבלה ראשית):** זה האינדקס הראשי שמפנה אותך לדף של השכונה המתאימה.

הדרישה שהטבלה תיכנס ב"דף אחד" אומרת שהמפה של השכונה לא יכולה להיות גדולה יותר מדף נייר סטנדרטי.

### פתרון צעד-אחר-צעד:

**א. מציאת גודל הדף (והאופסט):**
במבחן כזה, אם לא נתון גודל הדף, ננסה את הסטנדרט: **4KB**.
*   $4KB = 2^{12}$ בתים.
*   מסקנה: **Offset = 12 סיביות**.

**ב. חישוב מספר הכניסות בטבלה אחת:**
כמה שורות (PTE) של 4 בתים נכנסות בדף אחד של 4096 בתים?
*   $4096 / 4 = 1024$ שורות.
*   $1024 = 2^{10}$.
*   מסקנה: כל רמה בטבלה צריכה **10 סיביות** כדי להצביע על השורה הנכונה.

**ג. בדיקת הסיכום:**
*   רמה 1 (10) + רמה 2 (10) + אופסט (12) = **32 סיביות**.
*   התוצאה מתאימה בדיוק לנתון השאלה!

**תשובה סופית:** החלוקה היא **10-10-12**.

## 3. שאלה 2 ממבחן: תרגום כתובת וזיהוי Page Fault

**תיאור השאלה:**
נתונה מערכת עם כתובת של **15 סיביות**. גודל דף הוא **4KB**. 
נתונה טבלת דפים קטנה:
*   דף 0: Frame 1, Valid 1
*   דף 1: Frame 2, Valid 1
*   דף 2: Frame 0, Valid 0
*   דף 3: Frame 5, Valid 1

מה קורה כשמנסים לגשת לכתובת הווירטואלית הבינארית: `011000000001010`?

### סיפור ההדמיה: "הספרן והמחסן" 📚
התוכנה מבקשת מהספרן ספר. הכתובת שהיא נותנת מורכבת מ"מספר מדף" ו"מספר ספר במדף".
הספרן הולך לטבלה שלו. אם רשום שם "Valid 0", זה אומר שהספר בכלל לא בספרייה כרגע, הוא נמצא במחסן חיצוני (דיסק). הספרן צריך לעצור הכל וללכת להביא אותו. זה לוקח המון זמן.

### פתרון צעד-אחר-צעד:

**א. הפרדת הכתובת:**
1.  גודל דף 4KB ($2^{12}$) אומר שה-**Offset הוא 12 הסיביות הימניות**.
2.  הכתובת שקיבלנו: `011` | `000000001010`.
3.  החלק השמאלי (Page Number) הוא `011`. בבינארית `011` = **דף מספר 3**.

**ב. בדיקה בטבלה:**
1.  ניגש לשורה של דף 3 בטבלה שנתנו לנו.
2.  אנחנו רואים: `Frame 5, Valid 1`.
3.  מכיוון ש-Valid הוא 1, הכתובת נמצאת ב-RAM!

**ג. יצירת הכתובת הפיזית:**
1.  מספר המסגרת הוא 5 (בבינארית של 3 סיביות: `101`).
2.  נצמיד אליו את האופסט המקורי (שלא משתנה): `101` | `000000001010`.

**תשובה סופית:** הגישה הצליחה (אין Page Fault), והכתובת הפיזית היא המסגרת ה-5 באופסט המקורי.

## 4. טיפים ודברים שחוזרים על עצמם (שאלה 8)

1.  **חוק שימור האופסט:** האופסט (החלק הימני של הכתובת) הוא תמיד "קדוש". הוא עובר כמו שהוא מהכתובת הווירטואלית לפיזית. רק המספר משמאל משתנה.
2.  **נוסחת הזהב למספר כניסות:** אם מבקשים שטבלה תיכנס בדף, הנוסחה היא תמיד: `(Page Size) / (PTE Size)`.
3.  **כמה דפים תופסת הטבלה?** 
    *   אם התהליך משתמש ב**כל** הזיכרון: יש טבלה ראשית אחת + לכל שורה בה יש טבלה משנית. 
    *   *דוגמה:* בטבלה של 10-10-12, יהיו $1 + 1024 = 1025$ דפים.
4.  **זיהוי Page Fault:** אם ה-Valid bit הוא 0, אל תנסה לחשב כתובת פיזית. התשובה היא פשוט "Page Fault - הנתון יובא מהדיסק".
5.  **יחידות מידה:** במבחן אוהבים לערבב ביטים ובתים. זכור: 8 סיביות (bits) = בית אחד (Byte). אם הכתובת היא 32 סיביות, היא תופסת 4 בתים.
